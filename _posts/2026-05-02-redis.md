# Redis

Remote Dictionary Server의 줄임말로 In-memory Key-Value 저장소

쉽게 얘기해서 Key-Value 형태로 데이터를 저장하고 가져올 수 있는 초고속 저장소다

## Key-Value

앞서 얘기했듯, Redis는 데이터를 Key-Value 방식으로 저장한다

그러나, RDB처럼 테이블이란 개념 없이 바로 Key로 데이터를 접근한다

```
# MySQL 방식
+-----+----------+-----+
| id  | name     | age |
+-----+----------+-----+
| 123 | 홍길동   | 30  |
+-----+----------+-----+

SELECT name FROM users WHERE id = 123;

# Redis 방식
GET user:123:name
-> 결과: "홍길동"
```

## In-memory

Redis는 데이터를 메모리 즉, RAM에 저장한다

일반적인 데이터베이스인 MySQL이나 PostgreSQL은 데이터를 디스크에 저장한다

디스크 기반 저장소에서는 데이터를 읽을 때, 다음과 같은 과정을 거친다

```
1. CPU가 "id=123인 사용자 정보 가져와"라고 명령
2. 디스크에서 데이터를 찾음 (느림)
3. 디스크에서 RAM으로 데이터를 복사
4. RAM에서 CPU로 데이터 전달
5. CPU가 처리
```

반면, 레디스는 RAM에 저장하기 때문에 디스크를 거치지 않아서 디스크 기반 저장소보다도 훨씬 빠르다

```
1. CPU가 "user:123 가져와"라고 명령
2. RAM에서 바로 데이터 가져옴 (빠름)
3. CPU로 전달
4. CPU가 처리
```

그런데 인메모리 방식이 빠르다고 해서 무조건 좋은것은 아니다

RAM은 휘발성인 특징이 있기 때문에, 전원이 꺼지면 데이터가 전부 사라진다

물론 이런 단점을 극복하기 위한 몇가지 방법이 있다

RDB(스냅샷)이나 AOF(로그) 같은 기능으로 디스크에 백업을 한다거나, 복제본을 만들거나 하는 방법으로 말이다

또한, 용량이 제한적이며 비싸다

RAM은 디스크보다도 훨씬 비싸기 때문에 무작정 용량을 늘리기도 쉽지 않다

그래서 Redis는 모든 데이터를 저장하기 보다는, 자주 사용하는 데이터나 빠른 처리가 필요한 데이터만

저장하는 용도로 많이 쓰인다

## 그래서 Redis는 왜 쓰는데?

물론 속도 때문이다

실시간 처리가 필요한 앱에서는 속도가 생명이다

예를 들어, 지도 어플을 통해서 길을 찾아가고 있다고 해보자

현재 내 위치를 빠르게 표시해줘야 사용자가 지도에 표시된 내 위치를 기반으로 길을 찾아갈텐데

속도가 너무 느려서 현재 내 위치와 지도에 표시된 위치가 다르다면? 사용자는 불편함을 겪을 것이고 지도 어플을 사용하지 않게 될 것이다

MySQL과 같은 저장소를 쓰게 된다면 매번 디스크를 거쳐야 하기 때문에 속도가 매우 느릴 것이다

단건 조회라면 몰라도 수백, 수천명이 동시에 접근하다 보면 속도는 당연히 느려질 수밖에 없다

그런데 Redis는 메모리에 저장하기 때문에 디스크에 접근하기 시간이 없어서 훨씬 더 빠르게 결과를 받아볼 수 있다

## Redis의 핵심 특징 - 싱글 스레드

Redis는 단일 스레드로 동작한다

이걸 이해하기 전에 먼저 MySQL의 구조부터 보자

MySQL은 멀티 스레드를 사용한다. 여러 사용자가 동시에 쿼리를 보내면, 각 쿼리를 여러 스레드가 동시에 처리한다

```
사용자 A: "상품 정보 가져와" → 스레드 1이 처리
사용자 B: "주문 내역 가져와" → 스레드 2가 처리
사용자 C: "결제 처리해" → 스레드 3이 처리

→ 동시에 처리됨
```

이렇게 하면 여러 작업을 동시에 처리할 수 있다는 장점이 있지만, Race Condition이 발생한다는 단점도 있다

멀티 스레드에서 발생하는 Race Condition은 동시에 같은 데이터를 수정하려고 할 때 발생하는 문제다

```
시간 0초:
스레드 A (ATM 출금): 잔액 읽기 → 10,000원
스레드 B (온라인 결제): 잔액 읽기 → 10,000원

시간 1초:
스레드 A: 10,000 - 5,000 = 5,000원 계산
스레드 B: 10,000 - 3,000 = 7,000원 계산

시간 2초:
스레드 A: 잔액을 5,000원으로 저장
스레드 B: 잔액을 7,000원으로 저장 (A의 저장을 덮어씀!)

최종 잔액: 7,000원
```

물론 이를 방지하기 위해 Lock같은 방법을 사용해서 Race Condition을 방지할 수 있고 실제로 많이 쓰이는 방법이다

당연히 코드는 복잡해지고 성능도 떨어질 수밖에 없다

하지만 레디스는 스레드가 딱 1개다. 그래서 명령을 하나씩 순서대로 처리한다

따라서 Race Condition은 발생하지 않는다

또한 하나의 스레드만 있기 때문에 컨텍스 스위칭 비용도 없고, Lock을 쓰지 않기 때문에 Lock 대기 시간도 없다

물론, 스레드가 하나이기 때문에 오래 걸리는 작업이 있다면, 이 작업 하나 때문에 다른 작업이 대기하는 시간이 길어지기도 한다

따라서, 본인의 상황에 따라, 목적에 따라 Redis를 써야할지, DB를 써야할지 판단해야 한다

## NoSQL

Redis를 설명하기 위해서는 NoSQL에 대한 개념도 알아야 한다

SQL은 Structured Query Language의 약자로 "구조화된 질의 언어"라는 뜻이다

MySQL, PostgreSQL같은 관계형 데이터베이스에서 사용하는 언어다

관계형 데이터베이스는 데이터를 테이블에 저장하고, 테이블은 행과 열로 이루어져 있으며, 미리 정해진 구조(스키마)가 반드시 필요하다

```
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50) NOT NULL,
    age INT,
    email VARCHAR(100)
);
```

위의 코드처럼 미리 내가 어떤 이름의 테이블명을 쓸 것인지, 어떤 컬럼과 타입이 필요한지 미리 정의해야만 한다

데이터를 추가할 때도, 갱신할 때도 미리 구조를 정의해야 한다

반면, NoSQL은 Not Only SQL의 약자로 SQL만 사용하는게 아니라는 뜻이다. 더 정확히는 비관계형 데이터베이스를 의미한다

Redis는 NoSQL 데이터베이스로 스키마가 없다. 스키마가 없다는 것은 미리 구조를 정의할 필요 없이 데이터를 그냥 저장하면 된다는 뜻이다

```
# 바로 저장 가능
SET user:123 "홍길동"

# 나중에 더 많은 정보 저장하고 싶으면
HSET user:123 name "홍길동"
HSET user:123 age 30
HSET user:123 email "hong@example.com"

# 전화번호 추가? 그냥 바로 추가
HSET user:123 phone "010-1234-5678"
```

**그럼 NoSQL이 SQL보다 무조건 좋은게 아닌가?**

SQL만의 강점이 있다. 바로 JOIN이다

SQL에서는 JOIN을 통해 한 번의 쿼리로 여러 테이블의 데이터를 합쳐서 가져올 수 있다

그런데 Redis는 JOIN이 없다

여러 데이터를 합쳐서 보고 싶다면, 애플리케이션 레벨에서 각각 조회한 후 코드로 합쳐야 한다

따라서 SQL을 쓸 때는

- 데이터 간 관계가 중요할 때
- JOIN이 많이 필요할 때
- 복잡한 쿼리가 필요할 때
- 데이터 정합성이 최우선일 때

Redis를 쓸 때는

- 속도가 최우선일 때
- 단순한 조회가 대부분일 때
- 임시 데이터(캐시, 세션 등)
- 실시간 처리가 필요할 때

필요에 따라, 상황에 따라 적절한 저장소를 쓰면 된다

## Persistence

Redis는 인메모리 방식이라서 서버가 재시작되면 사라진다

그럼 진짜 중요한 데이터는 Redis에 저장하지 못하는 걸까?

아니다. Redis는 Persistence(영속성) 기능을 제공한다 즉, 데이터를 디스크에 저장하는 방식으로 영속성 기능을 제공한다

1. RDB(Redis Database)

RDB는 스냅샷 방식이다. 특정 시점의 데이터를 통째로 디스크에 저장하는 기능이다

```
현재 시각: 12:00
  ↓
메모리 상태를 통째로 디스크에 저장 (dump.rdb 파일)
  ↓
12:01, 12:02, ... 계속 작업
  ↓
15:00 (3시간 후)
  ↓
다시 스냅샷 저장
```

이런 RDB는 파일 크기가 작고 Redis 성능에도 별 영향을 끼치지 못한다는 장점이 있다

하지만, 마지막 스냅샷 이후 데이터는 여전히 유실된다는 단점이 있다

```
12:00 - 스냅샷 저장
12:01 - 데이터 추가
12:02 - 데이터 추가
12:03 - 데이터 추가
12:04 - 서버 다운! (스냅샷 전)

복구하면:
  ↓
12:00 스냅샷으로 복구
  ↓
12:01~12:04 데이터 유실
```

2. AOF(Append Only File)

AOF는 로그 방식이다. 모든 쓰기 명령을 하나하나 파일에 기록하는 방식이다

```
SET user:123 "홍길동"
  ↓
파일에 기록: SET user:123 "홍길동"

HSET order:1 amount 50000
  ↓
파일에 추가: HSET order:1 amount 50000

DEL temp:key
  ↓
파일에 추가: DEL temp:key
```

복구할 때는 이 파일을 처음부터 끝까지 재실행한다

```
서버 재시작
  ↓
AOF 파일 읽기
  ↓
SET user:123 "홍길동" 실행
HSET order:1 amount 50000 실행
DEL temp:key 실행
  ↓
데이터 완벽 복구
```

모든 명령을 기록하기 때문에 정확하고 데이터 유실이 거의 없다는 장점이 있지만

모든 로그를 기록하다보니 파일 크기가 매우 커질 수 있고, 성능에도 영향을 미친다

<br>

두 가지 방식 중 하나만 쓰는 것이 아니라, 두 가지를 모두 다 쓰는 것을 권장한다

평소에는 AOF로 안전하게 백업하고, 복구할 때는 RDB로 빠르게 복구한다

## Eviction Policy(데이터 삭제 정책)

메모리는 빠르지만, 크기가 매우 작은 저장소다

크기가 매우 적기 때문에 수많은 데이터를 저장하다 보면 저장소의 용량이 금방 꽉 찰 것이고, 용량이 꽉 차면 더 이상 저장할 수 없다

따라서, 이런 상황이 발생하기 전에 Redis는 메모리가 꽉차면 데이터 삭제 정책에 따라 데이터를 삭제한다

<br>

1. noeviction (기본값)

```
메모리 꽉 참
  ↓
새로운 데이터 저장 시도
  ↓
에러 발생
  ↓
저장 실패
```

아무것도 삭제하지 않고 그냥 에러를 낸다. 가장 안전하지만, 서비스가 멈출 수 있다

<br>

2. allkeys-lru

```
메모리 꽉 참
  ↓
모든 키 중에서 가장 오래 사용하지 않은 키(LRU) 삭제
  ↓
공간 확보
  ↓
새 데이터 저장
```

LRU(Least Recently Used)는 가장 최근에 사용되지 않는 것을 삭제하는 알고리즘이다

```
현재 시각: 12:00

user:1 - 마지막 접근: 11:50 (10분 전)
user:2 - 마지막 접근: 11:59 (1분 전)
user:3 - 마지막 접근: 10:00 (2시간 전) ← 가장 오래됨

메모리 부족
  ↓
user:3 삭제 (가장 오래 사용 안 함)
```

<br>

3. volatile-lru

TTL이 설정된 키 중에서만 LRU로 삭제하는 방식이다

TTL은 Time To Live의 약자로 유효시간을 말한다

유효시간이 만료되면, 자동으로 삭제되는 기능이다

TTL 명령어는

```
# 초 단위
SETEX user:123 3600 "홍길동"  # 3600초(1시간) 후 삭제

# 밀리초 단위
PSETEX user:123 5000 "홍길동"  # 5000밀리초(5초) 후 삭제

# 이미 있는 키에 TTL 추가
SET user:123 "홍길동"
EXPIRE user:123 3600  # 1시간 후 삭제
```

<br>

4. allkeys-lfu

LFU(Least Frequently Used)는 가장 적게 사용된 것을 삭제한다

```
user:1 - 접근 횟수: 100회
user:2 - 접근 횟수: 5회 ← 가장 적게 사용
user:3 - 접근 횟수: 50회

메모리 부족
  ↓
user:2 삭제 (가장 적게 사용)
```

<br>

5. allkeys-random

그냥 랜덤하게 삭제시킨다

<br>

6. volatile-ttl

TTL이 짧게 남은 키부터 삭제한다

7. 영구저장

물론 삭제하지 않고 영구적으로 저장하는 기능도 있다

```
PERSIST user:123
```

## 분산 락(Distributed Lock)

먼저 Lock이 뭔지부터 이해해보자

락은 잠금이라는 뜻으로, 여러 사람이 동시에 같은 자원을 수정하지 못하게 막는 장치다

아까 위에서 얘기했던 Race Condition을 방지하는 방법으로 Lock을 얘기했었는데, 바로 이 Lock이 동시에 같은 자원을 변경하는 것을 막는다

그럼 분산이라는 건 뭘 얘기하는 걸까?

분산은 여러 서버가 있다는 뜻이다

서비스가 커지면 서버를 여러 대 운영하게 된다

각 서버는 독립적으로 서버의 각 자원들을 다른 서버와 공유되지 않는다

그런데 각 서버에 락을 저장해봤자 소용이 없다. 서버 1이 락을 걸어도 서버 2는 알지 못하기 때문이다

이걸 해결하기 위해 중앙 저장소에 락을 저장한다

```
서버 1 ─┐
서버 2 ─┼─→ Redis (공유 저장소)
서버 3 ─┘
```

Redis에 락을 저장하면 모든 서버가 같은 락을 보게 된다

Redis에서 분산락을 사용하기 위한 명령어로 SETNX 명령을 사용한다

SET if Not eXists의 약자로 키가 없을 때만 값을 저장한다

```
SETNX lock:order:123 "server-1"
# 결과: 1 (성공, 키가 없어서 저장됨)

SETNX lock:order:123 "server-2"
# 결과: 0 (실패, 이미 키가 있음)
```

이걸 이용해서 락을 구현할 수 있게 된다

```
서버 1:
  SETNX lock:order:123 "server-1"
  → 1 반환 (성공)
  → "내가 락을 획득했다!"
  → 주문 처리
  → DEL lock:order:123 (락 해제)

서버 2 (거의 동시):
  SETNX lock:order:123 "server-2"
  → 0 반환 (실패, 서버 1이 이미 락을 가짐)
  → "락을 얻지 못했다"
  → 처리 포기 또는 대기

서버 3 (거의 동시):
  SETNX lock:order:123 "server-3"
  → 0 반환 (실패)
  → 처리 포기 또는 대기
```

자바에서는 아래처럼 코드로 구현한다

```
public boolean acceptOrder(String orderId) {
    String lockKey = "lock:order:" + orderId;
    
    // 락 획득 시도 (30초 TTL)
    Boolean acquired = redis.setNX(lockKey, "my-server-id", Duration.ofSeconds(30));
    
    if (!acquired) {
        // 락 획득 실패 (다른 서버가 이미 처리 중)
        return false;
    }
    
    try {
        // 락 획득 성공 → 주문 처리
        Order order = orderRepository.findById(orderId);
        
        if (order.getStatus() != OrderStatus.PENDING) {
            return false;  // 이미 처리됨
        }
        
        order.accept(riderId);
        orderRepository.save(order);
        
        return true;
        
    } finally {
        // 락 해제
        redis.delete(lockKey);
    }
}
```

그런데 자바 코드를 보면 Duration.ofSeconds(30)로 TTL을 30초로 설정했다

TTL을 꼭 설정해줘야 할까?

만약 TTL이 없다면

```
서버 1:
  SETNX lock:order:123 "server-1"
  → 성공
  → 주문 처리 시작
  → 처리 중 서버 갑자기 다운! (네트워크 끊김, 장애 등)
  → finally 블록 실행 안 됨
  → 락 해제 못 함

Redis:
  lock:order:123 = "server-1"  (영원히 남아있음)

서버 2, 3:
  SETNX lock:order:123 ...
  → 계속 실패
  → 영원히 락 획득 못 함
  → 주문 처리 불가능
```

데드락이 발생한다. 락을 잡은 서버가 죽어서 락을 해제하지 못하면, 그 락은 영원히 풀리지 않는다

```
서버 1:
  SETNX lock:order:123 "server-1" (TTL 30초)
  → 성공
  → 서버 다운!

30초 후:
  Redis가 자동으로 lock:order:123 삭제

서버 2:
  SETNX lock:order:123 "server-2"
  → 성공 (락 획득)
  → 주문 처리
```

TTL을 쓰면 최악의 경우 30초를 기다려야 하지만, 영원히 막히는 것보다는 낫다

**Redlock**

더 안전한 방법으로는 Redlock이라는 기능이 있다

우선 SETNX 방식은 다음과 같은 문제가 있을 수 있다

1. 타이밍 이슈

```
서버 1:
  작업 시간이 예상보다 오래 걸림
  → 20초 걸릴 줄 알았는데 40초 걸림
  → 30초 후 TTL 만료로 락 자동 해제

서버 2:
  30초 후 SETNX 성공 (락 획득)
  → 작업 시작

서버 1:
  아직 작업 중 (40초 중 35초 경과)
  → 서버 2와 동시에 같은 데이터 수정
  → 충돌!
```

2. Redis 단일 장애점

Redis 서버가 1대면, 그 서버가 죽으면 락 시스템 전체가 마비된다

<br>

Redlock 알고리즘은 Redis 창시자가 제안한 더 안전한 분산 락 알고리즘이다

```
Redis 서버를 5대 운영 (홀수 개)

락 획득:
  5대 중 3대 이상(과반수)에서 SETNX 성공해야 락 획득

락 해제:
  모든 서버에서 락 삭제
```

이렇게 하면, 1~2대 서버가 죽어도 괜찮고 타이밍 이슈도 줄어든다

단점으로는 복잡한 구현, 그리고 Redis 서버가 많이 필요하기 때문에 비용도 증가한다

대부분 시스템에서는 SETNX + TTL로도 충분하지만, 금융 같은 초정밀 시스템에서는 Redlock을 고려해봐야 한다

**분산 락 주의사항**

1. TTL은 작업 시간보다 충분히 길게 설정한다

```
// 나쁜 예
redis.setNX(lockKey, value, Duration.ofSeconds(5));  // 5초
processOrder();  // 10초 걸림 → 중간에 락 풀림

// 좋은 예
redis.setNX(lockKey, value, Duration.ofSeconds(30));  // 30초
processOrder();  // 10초 걸림 → 안전
```

2. finally에서 반드시 락 해제

```
try {
    // 작업
} finally {
    redis.delete(lockKey);  // 반드시!
}
```

3. 본인이 건 락만 해제하기

```
// 나쁜 예
redis.delete(lockKey);  // 다른 서버가 건 락을 삭제할 수도 있음

// 좋은 예
String myId = UUID.randomUUID().toString();
redis.setNX(lockKey, myId, ...);

// 해제 시
String currentValue = redis.get(lockKey);
if (myId.equals(currentValue)) {
    redis.delete(lockKey);  // 내가 건 락만 삭제
}
```

4. 스핀락 주의

```
// 나쁜 예: 계속 루프 돌면서 CPU 낭비
while (true) {
    if (redis.setNX(lockKey, value)) {
        break;
    }
    // 바로 다시 시도
}

// 좋은 예: 실패하면 잠시 대기
int retries = 0;
while (retries < 10) {
    if (redis.setNX(lockKey, value)) {
        break;
    }
    Thread.sleep(100);  // 100ms 대기
    retries++;
}
```

스핀락은 락을 얻을 때까지 계속 반복해서 시도하는 방식이다

**분산락은 꼭 써야하나?**

만약 서버 1대를 운영한다면 쓸 필요가 없다. 분산락은 서버가 여러 대 일 경우, 동시에 접근해 수정하지 못하도록 막기 위한 기술이기 때문

하지만, 현대 서비스에서는 트래픽 증가에 대비해 스케일 아웃이 필요하고, 무중단 배포 시에도 일시적으로 여러 인스턴스가 동시에 실행된다

따라서 서버가 현재 1대더라도 향후 확장성을 고려해 분산락을 사용하는 것이 일반적이다

## 자료구조별 내부 동작 원리

Redis는 겉보기엔 단순하지만, 내부적으로는 성능을 위해 여러 최적화를 한다

**String의 내부 구조**

Redis 명령에서는 모든 값이 문자열로 전달된다

값을 받으면 Redis가 내부적으로 타입을 판단한다

String이 가장 단순해 보이지만, 실제로는 3가지 방식으로 저장된다

1. int

값이 정수고, long 범위 내라면 실제로 정수로 저장한다

```
SET counter 123

내부적으로:
  type: int
  value: 123 (정수)
```

정수로 저장하기 때문에 메모리를 적게 쓴다

2. embstr(짧은 문자열)

문자열이 44바이트 이하면 embstr 방식으로 저장한다

```
SET user:name "홍길동"

내부적으로:
  type: embstr
  value: "홍길동" (연속된 메모리 공간)
```

메모리 할당을 한 번만 하기 때문에 효율적이다

이게 무슨소리냐면

프로그램이 데이터를 저장하려면 메모리 공간을 받아야하고, 이 과정을 메모리 할당이라고 한다

그런데 메모리 할당은 비싼 작업이다

```
1. OS에게 "메모리 주세요" 시스템 콜
2. OS가 빈 공간 검색
3. 메타데이터 기록 (크기, 위치 등)
4. 주소 반환
```

시간이 걸리는 작업이고, 특히 여러 번 할당하면 더 느려진다

아래에서도 얘기하겠지만, raw는 메모리 할당을 두 번하지만 embstr은 딱 한 번만 메모리를 할당하기 때문에 효율적이라고 말하는 것이다

```
1차 할당만: Object + 문자열을 연속된 공간에 함께 저장
┌─────────────────────────────────────┐
│ Redis Object                        │
│ - type: string                      │
│ - encoding: embstr                  │
│ "홍길동" ← 바로 여기 저장!           │
└─────────────────────────────────────┘

메모리 주소 0x1000: 
  [Redis Object 메타데이터 16바이트]
  [문자열 "홍길동" 9바이트]
  
→ 연속된 25바이트를 한 번에 할당
```

3. raw (긴 문자열)

문자열이 44바이트 초과라면 raw 방식으로 저장한다

```
SET article:content "엄청 긴 글 내용..."

내부적으로:
  type: raw
  value: 포인터 → 별도 메모리 공간
```

이런식으로 데이터에 따라 작은 값은 간단하게, 큰 값은 유연하게 관리해 최적화한다

embstr과는 다르게 메모리를 두 번 할당한다

```
1차 할당: Redis Object 구조체 저장
┌─────────────────────┐
│ Redis Object        │
│ - type: string      │
│ - encoding: raw     │
│ - ptr: 0x1234 ───┐  │  (포인터)
└──────────────────┼──┘
                   │
                   │  2차 할당: 실제 문자열 저장
                   └──> ┌───────────────────┐
                        │ "안녕하세요..."    │
                        └───────────────────┘

메모리 주소 0x1000: Redis Object (약 16바이트)
  ↓
메모리 주소 0x1234: 실제 문자열 "안녕하세요..." (48바이트)
```

이렇게 두번에 나눠서 메모리를 할당하는 이유는

문자열이 너무 길어서 한 번에 못담기 때문이다

구체적인 예시를 보자

```
# embstr 방식(44바이트 이하)

초기 상태:
┌──────────────────────────────┐
│ Object | "hello"             │  64바이트 할당
└──────────────────────────────┘

문자열 수정 시도: "hello" → "hello world"
  ↓
64바이트 안에 안 들어감!
```

이러면 어떻게 해야 할까?

```
방법 1: 그냥 뒤에 추가?
  → 불가능 (이미 다른 데이터가 있을 수 있음)

방법 2: 전체 재할당?
  ┌──────────────────────────────┐
  │ Object | "hello"             │  기존 (버림)
  └──────────────────────────────┘
  
  ┌──────────────────────────────────────┐
  │ Object | "hello world"               │  새로 할당 (128바이트)
  └──────────────────────────────────────┘
  
  → 비효율적 (전체를 복사해야 함)
```

Redis에는 다음과 같은 방법으로 해결한다

```
SET key "hello"  # embstr로 저장

APPEND key " world"  # 문자열 추가 시도
  ↓
embstr → raw로 타입 변경
  ↓
새로 할당


초기 상태:
┌──────────────┐
│ Object       │  16바이트
│ ptr ─────┐   │
└──────────┼───┘
           │
           └──> ┌─────────────────────┐
                │ "긴 문자열..."       │  100바이트
                └─────────────────────┘

문자열 수정: "긴 문자열..." → "더 긴 문자열..."
  ↓
Object는 그대로 두고
  ↓
문자열 영역만 재할당
  ↓
┌──────────────┐
│ Object       │  그대로 (16바이트)
│ ptr ─────┐   │
└──────────┼───┘
           │
           └──> ┌──────────────────────────┐
                │ "더 긴 문자열..."         │  새로 할당 (150바이트)
                └──────────────────────────┘
```

Object는 건드리지 않고 문자열만 재할당하고 포인터만 바꾸면 되기 때문에 효율적이다

<br>

**List의 내부 구조**

List는 예전에는 LinkedList와 ZipList 두 가지를 섞어 썼다

LinkedList 구조

```
[node1] → [node2] → [node3] → [node4]

각 노드마다:
- 이전 노드 포인터
- 다음 노드 포인터
- 값
```

앞뒤 추가나 삭제가 O(1)로 매우 빠르지만, 포인터 때문에 메모리를 많이 쓴다

<br>

ZipList 구조

```
[전체 크기][길이][entry1][entry2][entry3][끝 표시]

연속된 메모리 블록
```

메모리는 효율적이지만 중간 삽입/삭제는 O(N)으로 느리다

요소가 적으면 ZipList, 많아지면 LinkedList로 전환하는 전략을 썼었지만,

Redis 7.0 이상부터는 ListPack이라는 새로운 구조로 통일했다

ZipList의 개선 버전으로, 메모리 효율적이면서도 더 빠르다

<br>

**Hash의 내부 구조**

해시도 크기에 따라 다르게 저장된다

작은 해시 (기본값 512개 이하, 64바이트 이하)

```
[전체 크기][길이][entry1][entry2][entry3][끝 표시]

연속된 메모리 블록
```

큰 해시

```
Hash Table:
  해시 함수로 버킷 선택
  버킷에 field-value 저장
```

여기서 버킷이란 데이터를 저장하는 칸을 말한다

해시 테이블은 여러 개의 버킷으로 이루어진 배열이다

<br>

예시

```
HSET user:123 name "홍길동"
HSET user:123 age 30
HSET user:123 email "hong@example.com"

내부적으로 (작은 Hash):
  [name]["홍길동"][age][30][email]["hong@example.com"]
  
  → HGET user:123 age 하면?
  → 선형 탐색: name, age 찾음 → 30 반환
  → O(N)이지만 N이 작아서 빠름

# Hash가 커지면

# 필드가 1000개
HSET user:123 field1 value1
HSET user:123 field2 value2
...
HSET user:123 field1000 value1000

내부적으로 Hash Table로 전환:
  해시 함수(field1) → 버킷 10 → value1
  해시 함수(age) → 버킷 523 → 30
  
  → HGET user:123 age 하면?
  → 해시 함수 계산 → 버킷 직접 접근
  → O(1)

# 더 쉽게 예를 들어보면

버킷:
[0] [1] [2] [3] [4] [5] [6] [7]

30 찾기:
  hash(30) = 12345
  12345 % 8 = 1
  → 버킷 1로 바로 이동
  
→ O(1)
```

**Sorted Set의 내부 구조**

Sorted Set은 점수로 정렬되는 집합을 저장하는 자료구조다

Set은 순서가 없고 중복만 없는 구조라면, Sorted Set은 여기서 점수(score)가 추가된 개념이다

각 멤버(member)마다 점수(score)를 부여하고, 점수 순서대로 자동 정렬된다

```
# 점수와 함께 추가
ZADD ranking 100 "user1"
ZADD ranking 200 "user2"
ZADD ranking 150 "user3"

# Redis 내부에서 자동 정렬됨:
# user1: 100
# user3: 150
# user2: 200
```

점수가 낮은 것부터 높은 순서로 자동 정렬된다

Sorted Set은 Redis에서 가장 복잡한 자료구조다. 두 가지 구조를 동시에 사용한다

1. Skip List

```
레벨 3:  1 -----------------------> 50
레벨 2:  1 ------> 20 -----> 40 --> 50
레벨 1:  1 -> 10 -> 20 -> 30 -> 40 -> 50

점수 순서대로 정렬되어 있음
```

쉽게 얘기해서 정렬된 링크드 리스트에 고속도로를 추가한 것이다

검색으로 50을 찾으려고 할 때

```
레벨 3에서 50으로 점프 (O(log N))

# 만약 LinkedList였으면
1 → 10 → 20 → 30 → 40 → 50 (O(N))
```

그냥 쉽게 얘기해서 중간 지점을 두고, 그 중간 지점으로 한번에 이동할 수 있는 통로를 미리 만들어 두는 것이라 생각하면 된다

```
# 30을 찾는 과정을 보자

시작: 레벨 3 (최상층)
┌───────────────────────────┐
│ 레벨 3: 1 ─────────────→ 50 │
└───────────────────────────┘

1에서 시작
  ↓
다음은 50
  ↓
50 > 30 (너무 멀리 갔음)
  ↓
아래 레벨로 내려가기


레벨 2로 내려감
┌──────────────────────────┐
│ 레벨 2: 1 ──→ 20 ──→ 40 ──→ 50 │
└──────────────────────────┘

1에서 시작
  ↓
다음은 20
  ↓
20 < 30 (아직 덜 갔음)
  ↓
20으로 이동
  ↓
20에서 다음은 40
  ↓
40 > 30 (너무 멀리 갔음)
  ↓
아래 레벨로 내려가기


레벨 1로 내려감
┌─────────────────────────┐
│ 레벨 1: 1 → 10 → 20 → 30 → 40 → 50 │
└─────────────────────────┘

20에서 시작 (현재 위치)
  ↓
다음은 30
  ↓
30 == 30 (찾음!)

→ 총 3단계 (레벨 3 → 레벨 2 → 레벨 1)
```

2. Hash Table

member -> socre 매핑을 저장한다

즉, key-value 형태로 데이터를 저장해둔다

```
{
  "user1": 100,
  "user2": 200,
  "user3": 150
}
```

<br>

왜 이 두개를 같이 쓸까?

게임 랭킹을 예로 들어보자

```
ZADD ranking 100 "철수"
ZADD ranking 200 "영희"
ZADD ranking 150 "민수"
ZADD ranking 300 "지영"
```

데이터가 이렇게 있을 때 상위 10명을 보려면?

```
ZRANGE ranking 0 9  # 1등부터 10등까지

# 이때 skip list를 써서 가져온다 -> 이미 점수가 낮은순으로 정렬되어있기 때문에 앞에서부터 가져오면 된다
100(철수) → 150(민수) → 200(영희) → 300(지영)
```

그럼 이제 철수의 점수가 몇점인지 알려면?

```
ZSCORE ranking "철수"

# 이때 hash table을 사용해서 점수를 가져온다
"철수" → 100
"영희" → 200
"민수" → 150
"지영" → 300

"철수" 검색
→ 바로 100 찾음
```

Sorted Set는 점수와 관련된 데이터를 다루는 자료구조다

랭킹을 조회할 수도 있고, 특정 member의 score를 가져오고 싶을 때도 있다

이때, 상황에 맞게 skip list를 쓰거나 hash table을 써서 데이터를 빠르게 가져올 수 있도록 2개의 구조를 함께 쓰는 방식으로 만들어졌다

**GEO의 내부 구조**

GEO는 지리적 위치 정보를 다루는 자료구조다

경도와 위도 좌표를 저장하고, 거리 계산, 반경 검색 등을 할 수 있다

GEO는 사실 Sorted Set의 변형이다

지구 좌표(경도, 위도)를 Geohash라는 문자열로 변환한다

```
위도 37.4979, 경도 127.0276
  ↓ Geohash 변환
"wydm6"
  ↓ 숫자로 변환
3479099956230398
  ↓ Sorted Set에 저장 (score로 사용)
```

가까운 위치 = 비슷한 Geohash

즉, 위도, 경도를 문자열로 바꿔주는데 이때, 위도, 경도가 가까울 수록 비슷한 문자열로 변환해줘서
가까운 위치를 검색할 수 있게 해준다

```
강남역: "wydm6"
역삼역: "wydm7"  ← 비슷함
부산역: "wydjk"  ← 다름
```

반경 검색을 한다면

```
GEORADIUS riders 127.0280 37.4980 3 km

내부 동작:
1. 중심 좌표를 Geohash로 변환
2. 반경 3km를 Geohash 범위로 변환
3. Sorted Set에서 범위 조회 (ZRANGEBYSCORE)
4. 정확한 거리 계산 (필터링)
```
