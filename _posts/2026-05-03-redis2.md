# Redis Master-Slave Replication

## Master와 Slave가 뭔가?

**Master = 주인, 원본 데이터를 가진 서버**
**Slave = 종속, 원본을 복사한 서버**

Redis는 원래 하나의 서버로 동작한다. 그런데 서비스가 커지면 서버 하나로는 부족하다. 그래서 **여러 대의 Redis 서버를 함께 사용**하는데, 이때 서버들의 역할을 나눈다. 그게 바로 Master와 Slave다.

**Master는 "원본"을 관리하는 서버**다. 모든 데이터의 기준이 되는 서버다.

**Slave는 "복사본"을 가진 서버**다. Master의 데이터를 그대로 복사해서 가지고 있다.

---

## 왜 역할을 나눠야 하나?

Redis 서버가 1대만 있을 때의 문제점을 먼저 이해해야 한다.

### 문제 1: 성능 한계

**상황:**
배달 서비스를 운영한다고 가정해보자. Redis 서버가 1대만 있다.

```
Redis 서버 1대
```

이 서버는 모든 요청을 혼자 처리해야 한다:

**쓰기 요청 (데이터 변경):**
- 배달기사가 5초마다 위치 업데이트: `GEOADD riders 127.0276 37.4979 "rider:1"`
- 주문이 들어옴: `SET order:123 "..."`
- 주문 상태 변경: `HSET order:123 status "배달중"`

**읽기 요청 (데이터 조회):**
- 사용자가 반경 3km 배달기사 검색: `GEORADIUS riders 127.0280 37.4980 3 km`
- 주문 정보 조회: `GET order:123`
- 배달 현황 조회: `HGET order:123 status`

**문제:**

배달 서비스에서는 **읽기 요청이 압도적으로 많다**. 왜냐하면:
- 배달기사는 100명
- 사용자는 10,000명

배달기사 100명이 5초마다 위치를 보내니까 쓰기 요청은 초당 20건 정도다.

하지만 사용자 10,000명이 배달 현황을 확인하니까 읽기 요청은 초당 수만 건이다.

```
초당 요청:
  쓰기: 20건
  읽기: 50,000건
  ───────────────
  총: 50,020건

Redis 1대 한계: 약 100,000 QPS (Queries Per Second)
```

지금은 괜찮지만, 사용자가 더 늘어나면?

```
사용자 100,000명으로 증가:
  읽기: 500,000건
  
→ 서버 한계 초과
→ 느려짐
→ 타임아웃 발생
→ 서비스 장애
```

**핵심 문제:** 서버 1대로는 많은 읽기 요청을 감당할 수 없다.

---

### 문제 2: 장애 시 서비스 중단

**상황:**
Redis 서버 1대만 운영 중이다.

```
Redis 서버 1대
  ↓ (모든 데이터 여기만 있음)
```

갑자기 서버에 문제가 생긴다:
- 하드웨어 고장 (메모리 불량, 디스크 오류)
- 네트워크 장애
- 소프트웨어 버그로 인한 크래시
- 사람의 실수 (잘못된 명령 실행)

```
Redis 서버 다운!
  ↓
모든 데이터 접근 불가능
  ↓
배달 현황 조회 불가
  ↓
주문 처리 불가
  ↓
서비스 완전 중단
```

**복구 과정:**

Redis에는 백업 기능이 있다 (RDB, AOF). 하지만:

1. 서버를 다시 시작해야 한다 (수 분 소요)
2. 백업 파일을 로드해야 한다 (데이터 양에 따라 수 분~수십 분)
3. 마지막 백업 이후의 데이터는 유실될 수 있다

```
예:
  12:00 - 마지막 RDB 백업
  12:30 - 서버 다운
  
  복구:
    12:00 시점 데이터로 복구
    12:00~12:30 사이 데이터 유실
```

**핵심 문제:** 서버가 죽으면 서비스가 멈추고, 복구까지 시간이 오래 걸린다.

---

### 해결책: Master-Slave 구조

이 두 문제를 해결하기 위해 **여러 대의 서버를 사용하되, 역할을 나눈다**.

```
        Master (1대)
         /  |  \
        /   |   \
    Slave Slave Slave
     (1)   (2)   (3)
```

**Master의 역할:**
- 원본 데이터를 관리한다
- 주로 쓰기 작업을 담당한다
- 데이터가 변경되면 Slave들에게 알려준다

**Slave의 역할:**
- Master의 데이터를 복사해서 가진다
- 주로 읽기 작업을 담당한다
- Master가 보내주는 변경사항을 계속 반영한다

**장점 1: 성능 문제 해결**

```
쓰기 요청 → Master (1대가 처리)
읽기 요청 → Slave 1, 2, 3 (3대가 나눠서 처리)

읽기 성능 3배 향상!
```

Slave를 더 추가하면 읽기 성능은 계속 올라간다.

**장점 2: 장애 문제 해결**

```
Master 다운!
  ↓
Slave들은 살아있음
  ↓
Slave 중 하나를 새 Master로 승격
  ↓
서비스 계속 가능
```

데이터가 여러 서버에 복사되어 있으니 한 서버가 죽어도 괜찮다.

---

## Master는 꼭 쓰기만 해야 하나?

**아니다. Master도 읽기를 할 수 있다.**

이게 헷갈리는 부분인데, 차근차근 설명하겠다.

### Master가 할 수 있는 일

**1. 쓰기 작업 (필수)**

```bash
SET user:1 "철수"
INCR counter
HSET order:123 status "배달중"
```

Master는 유일하게 데이터를 변경할 수 있는 서버다. 모든 쓰기 작업은 반드시 Master에서만 가능하다.

**2. 읽기 작업 (가능)**

```bash
GET user:1
ZRANGE ranking 0 9
HGET order:123 status
```

Master도 읽기를 할 수 있다. 기술적으로 막혀있지 않다.

---

### 그런데 왜 Master에서 읽기를 안 할까?

Master는 이미 할 일이 많기 때문이다.

**Master가 해야 하는 일들:**

1. **쓰기 요청 처리**
   ```
   클라이언트로부터 SET, INCR, DEL 등의 명령 받아서 처리
   ```

2. **데이터 영속화 (Persistence)**
   ```
   RDB 스냅샷 생성 (주기적으로)
   AOF 파일에 명령 기록 (매 명령마다 또는 1초마다)
   ```
   
   이 작업들은 디스크 I/O를 발생시켜서 부담이 된다.

3. **Slave로 데이터 복제**
   ```
   모든 쓰기 명령을 Slave들에게 전송
   Slave가 연결되면 RDB 파일 생성해서 전송
   ```
   
   Slave가 3대면 3대 모두에게 명령을 보내야 한다.

**만약 Master에서 읽기까지 한다면:**

```
Master의 부하:
  쓰기 처리
+ 디스크 저장
+ Slave 복제
+ 읽기 처리  ← 추가!
─────────────
과부하 위험
```

Master가 과부하 상태가 되면:
- 쓰기 처리가 느려진다
- Slave 복제가 지연된다
- 최악의 경우 Master가 다운된다

**그래서 실무에서는:**

```
Master: 쓰기만 담당 (부담을 줄임)
Slave: 읽기만 담당 (Master를 보호)
```

이렇게 역할을 명확히 나눈다.

---

### 하지만 반드시 이렇게 해야 하는 건 아니다

**경우에 따라 다르다.**

**Case 1: Slave가 없는 경우**

```
Redis 서버 1대 (Master만 있음)
```

Slave를 안 둘 수도 있다. 트래픽이 적은 소규모 서비스라면 굳이 Slave를 둘 필요가 없다.

이 경우 Master는:
```
쓰기 + 읽기 모두 처리
```

당연하다. 서버가 1대뿐이니까.

---

**Case 2: Slave는 백업용으로만**

```
애플리케이션:
  쓰기 → Master
  읽기 → Master (Slave 안 씀!)

Slave:
  Master 데이터 복사만 (백업 목적)
  읽기 요청 안 받음
```

이것도 가능하다. Slave를 장애 대비용 백업으로만 사용하는 것이다.

트래픽이 적어서 Master 1대로 충분하면 이렇게 운영할 수 있다.

---

**Case 3: 읽기 부하 분산 (가장 일반적)**

```
애플리케이션:
  쓰기 → Master
  읽기 → Slave 1, 2, 3 (로드밸런싱)

Master:
  쓰기만 담당
  
Slave:
  읽기만 담당
```

트래픽이 많아서 읽기 성능이 중요한 경우 이렇게 한다.

**코드 예시:**

```java
@Service
public class UserService {
    
    // Master용 Redis 연결
    @Autowired
    @Qualifier("masterRedis")
    private RedisTemplate<String, String> masterRedis;
    
    // Slave용 Redis 연결
    @Autowired
    @Qualifier("slaveRedis")
    private RedisTemplate<String, String> slaveRedis;
    
    // 사용자 정보 저장 (쓰기)
    public void saveUser(String userId, String name) {
        // Master에 저장
        masterRedis.opsForValue().set("user:" + userId, name);
    }
    
    // 사용자 정보 조회 (읽기)
    public String getUser(String userId) {
        // Slave에서 조회
        return slaveRedis.opsForValue().get("user:" + userId);
    }
}
```

이렇게 코드 레벨에서 Master와 Slave를 구분해서 사용한다.

---

**정리:**

Master는 읽기를 할 수 있다. 하지만:
- Master가 쓰기로 이미 바쁘면 읽기는 Slave에게 맡긴다
- 트래픽이 적으면 Master에서 읽기도 할 수 있다
- **"반드시"는 아니지만, 실무에서는 보통 Master는 쓰기만 담당한다**

---

## Slave는 쓰기가 절대 안 되나?

**기본적으로 안 된다. Slave는 읽기 전용이다.**

### Slave의 기본 설정

Redis를 Slave로 설정하면 자동으로 읽기 전용 모드가 된다.

**redis.conf (Slave 설정 파일):**

```conf
# Master 정보
replicaof 192.168.1.100 6379

# 읽기 전용 설정 (기본값)
replica-read-only yes
```

`replica-read-only yes`가 기본값이다. 이 설정이 켜져 있으면 Slave에서는 쓰기가 불가능하다.

**Slave에서 쓰기 시도:**

```bash
# Slave 서버에 접속
redis-cli -h 192.168.1.101 -p 6379

# 쓰기 명령 실행
SET user:1 "영희"
```

**에러 발생:**

```
(error) READONLY You can't write against a read only replica.
```

"읽기 전용 복제본에는 쓸 수 없다"라는 에러가 난다.

---

### 왜 Slave는 쓰기가 불가능한가?

**데이터 일관성**을 유지하기 위해서다.

**만약 Slave에서 쓰기가 가능하다면 어떻게 될까?**

**상황:**

```
Master:
  user:1 = "철수"

Slave 1:
  user:1 = "철수" (Master에서 복제됨)
```

여기서 누군가 Slave 1에 직접 접속해서 데이터를 변경한다:

```bash
# Slave 1에서
SET user:1 "영희"
```

**결과:**

```
Master:
  user:1 = "철수" (그대로)

Slave 1:
  user:1 = "영희" (변경됨!)
```

**문제 발생:**

1. **Master와 Slave의 데이터가 다르다**
   - Master: "철수"
   - Slave 1: "영희"
   - 어느 게 정답인가?

2. **복제가 계속되면서 덮어쓴다**
   
   Master에서 다른 필드를 변경한다:
   ```bash
   # Master에서
   SET user:2 "민수"
   ```
   
   이 명령이 Slave로 복제된다. 그런데 Master는 여전히 `user:1 = "철수"`를 가지고 있으니, 나중에 다시:
   ```bash
   # Master에서 (다시 설정하거나, 재시작 후 복제)
   SET user:1 "철수"
   ```
   
   이게 Slave로 가면:
   ```
   Slave 1:
     user:1 = "영희" → "철수"로 덮어씀
   ```
   
   Slave 1에서 변경한 내용이 사라진다!

3. **어느 서버에서 읽느냐에 따라 다른 값을 본다**
   
   ```
   사용자 A: Master에서 읽기 → "철수"
   사용자 B: Slave 1에서 읽기 → "영희"
   
   같은 데이터인데 다른 값!
   ```

**결론:**

Slave에서 쓰기를 허용하면 데이터가 뒤죽박죽이 된다. 그래서 **Slave는 읽기 전용**으로 만든다.

---

### 모든 쓰기는 Master로

**올바른 방식:**

```
모든 쓰기 요청은 Master로 보냄
  ↓
Master가 데이터 변경
  ↓
Master가 Slave들에게 변경사항 전송
  ↓
Slave들이 동일하게 변경
  ↓
모든 서버가 같은 데이터 유지
```

이렇게 하면 데이터 일관성이 보장된다.

---

### 예외: replica-read-only no

**기술적으로는 Slave에서 쓰기를 허용할 수 있다.**

**redis.conf 변경:**

```conf
replicaof 192.168.1.100 6379
replica-read-only no  ← 쓰기 허용
```

이렇게 설정하면 Slave에서도 쓰기가 가능해진다.

**하지만 절대 권장하지 않는다!**

**쓰는 경우:**

1. **임시 계산용 Slave**
   ```
   Master: 실제 데이터
   Slave: 복잡한 계산 전용 (임시 데이터 저장)
   
   Slave에서만 쓰는 임시 키:
     SETEX temp:calculation:123 60 "..."
   ```

2. **테스트 환경**
   ```
   운영과 무관한 테스트 데이터를 Slave에 직접 넣어봄
   ```

3. **Master와 무관한 로컬 캐시**
   ```
   Slave만의 로컬 캐시 키:
     SET local:cache:xyz "..."
   ```

**운영 환경에서는 절대 사용하지 마라!**

---

## replica-read-only는 자동으로 설정되나?

**자동으로 설정된다.**

### 기본 동작

Redis를 Slave로 설정하면:

```conf
# redis.conf
replicaof 192.168.1.100 6379
```

이것만 적으면 된다.

**Redis가 시작할 때:**

```
1. Redis 시작
2. redis.conf 읽기
3. "replicaof가 있네? 이 서버는 Slave구나"
4. replica-read-only 설정 확인
5. 설정이 없으면 기본값 적용: replica-read-only yes
6. 읽기 전용 모드로 시작
```

**즉, 따로 설정하지 않아도 자동으로 읽기 전용이 된다.**

---

### 확인 방법

**Slave가 제대로 읽기 전용인지 확인:**

```bash
redis-cli -h 192.168.1.101

# 설정 확인
CONFIG GET replica-read-only
```

**출력:**

```
1) "replica-read-only"
2) "yes"
```

또는:

```bash
INFO replication
```

**출력:**

```
# Replication
role:slave
master_host:192.168.1.100
master_port:6379
master_link_status:up
...
slave_read_only:1  ← 1이면 읽기 전용
```

---

### 명시적으로 적을 수도 있다

**더 명확하게 하고 싶으면:**

```conf
# redis.conf
replicaof 192.168.1.100 6379

# 읽기 전용 (명시적으로 적음)
replica-read-only yes
```

이렇게 적어도 되지만, 결과는 같다. 어차피 기본값이 `yes`니까.

---

### 쓰기를 허용하려면 명시적으로 변경

**쓰기를 허용하고 싶으면 (권장 안 함):**

```conf
replicaof 192.168.1.100 6379
replica-read-only no  ← 반드시 이렇게 명시해야 함
```

또는 런타임에:

```bash
redis-cli -h 192.168.1.101

CONFIG SET replica-read-only no
```

---

**정리:**

```
Slave로 설정하면:
  ↓
자동으로 replica-read-only yes 적용
  ↓
읽기 전용 모드

따로 설정 안 해도 Redis가 알아서 읽기 전용으로 만들어줌
```

## 복제 동작 원리 (정말 자세히)

Slave가 Master로부터 어떻게 데이터를 받는지 단계별로 설명하겠다.

### 1단계: Slave 시작 및 Master 연결

**설정:**

```conf
# slave.conf
port 6380
replicaof 192.168.1.100 6379
```

**Slave 시작:**

```bash
redis-server slave.conf
```

**내부 동작:**

```
1. Redis 프로세스 시작
   메모리 할당, 네트워크 포트 열기 등

2. 설정 파일 읽기
   "replicaof 192.168.1.100 6379" 발견

3. Master에 연결 시도
   TCP 연결: 192.168.1.100:6379

4. 연결 성공
   
5. Slave → Master로 PING 전송
   "살아있나요?"

6. Master → Slave로 PONG 응답
   "네, 살아있어요"

7. Slave → Master로 복제 요청
   "PSYNC ? -1"
   의미: "처음이에요, 전체 복제 주세요"
   
   - "?" = Replication ID 모름 (처음이니까)
   - "-1" = Offset 모름 (처음이니까)

8. Master → Slave로 응답
   "FULLRESYNC 8371b4fb1155b71f4a04d3e1bc3e18c4a990aeeb 0"
   의미: "알았어, 전체 동기화 할게. 내 ID는 8371...이고 offset은 0부터 시작이야"
```

PSYNC는 "Partial Sync"의 약자다. 하지만 처음이라 Full Sync를 한다.

---

### 2단계: Full Sync (전체 동기화)

**Full Sync가 필요한 이유:**

Slave가 처음 연결되면 아무 데이터도 없다. Master의 모든 데이터를 받아와야 한다.

**Master의 동작:**

```
1. 백그라운드에서 RDB 파일 생성 시작
   fork() 시스템 콜로 자식 프로세스 생성
   자식 프로세스가 RDB 파일 작성
   
2. RDB 생성 중에도 Master는 계속 서비스
   클라이언트 요청 처리 계속
   
3. 이 시점 이후의 모든 쓰기 명령을 버퍼에 저장
   (Replication Buffer)
   
   예:
   SET user:3 "민수"  → 버퍼에 저장
   INCR counter       → 버퍼에 저장
   DEL temp:key       → 버퍼에 저장

4. RDB 파일 생성 완료
   dump.rdb (예: 100MB)

5. RDB 파일을 네트워크로 Slave에 전송
   TCP 소켓을 통해 바이트 스트림 전송
```

**Slave의 동작:**

```
1. Master로부터 RDB 파일 수신 시작
   네트워크로 바이트 받기

2. 임시 파일로 저장
   temp-1234.rdb

3. 전체 수신 완료 확인

4. 기존 메모리 데이터 삭제 (있다면)
   FLUSHALL 수행

5. RDB 파일을 메모리로 로드
   파일 읽어서 파싱
   각 키-값을 메모리에 저장
   
   예:
   user:1 = "철수"
   user:2 = "영희"
   counter = 100
   ...

6. 로드 완료
```

**버퍼 전송:**

```
Master:
  "RDB 파일 보냈어. 그 사이에 변경된 것들 보낼게"
  
  Replication Buffer 내용 전송:
    SET user:3 "민수"
    INCR counter
    DEL temp:key

Slave:
  명령 하나씩 실행:
    SET user:3 "민수"  → 실행
    INCR counter       → 실행
    DEL temp:key       → 실행

완료:
  Master와 Slave가 완전히 동일한 데이터 보유
```

---

**시간 흐름으로 보면:**

```
시각 12:00:00.000
  Master: RDB 생성 시작
  
시각 12:00:00.100
  클라이언트: SET user:3 "민수"
  Master: 처리 + 버퍼에 저장
  
시각 12:00:00.200
  클라이언트: INCR counter
  Master: 처리 + 버퍼에 저장

시각 12:00:05.000
  Master: RDB 생성 완료 (5초 소요)
  Master: RDB 파일 Slave로 전송 시작

시각 12:00:10.000
  Slave: RDB 수신 완료 (5초 소요)
  Slave: 메모리에 로드 시작

시각 12:00:12.000
  Slave: 로드 완료 (2초 소요)

시각 12:00:12.001
  Master: 버퍼 내용 전송
  Slave: 버퍼 명령 실행

시각 12:00:12.010
  완료: 동기화 완료
```

총 12초 정도 걸렸다. 데이터가 많으면 더 오래 걸릴 수 있다.

---

### 3단계: Partial Sync (증분 동기화)

**Full Sync 이후:**

이제 Master와 Slave는 동일한 데이터를 가지고 있다. 앞으로는 변경사항만 전송하면 된다.

**Master에서 쓰기 발생:**

```
시각 12:00:13.000
  클라이언트 → Master: SET user:4 "지영"

Master:
  1. 명령 실행
     user:4 = "지영" 저장
  
  2. Slave로 명령 전송
     "SET user:4 지영" 전송
  
  3. Offset 증가
     Replication Offset: 12345 → 12400
```

**Slave의 동작:**

```
Slave:
  1. Master로부터 명령 수신
     "SET user:4 지영"
  
  2. 명령 실행
     SET user:4 "지영"
  
  3. Offset 증가
     Replication Offset: 12345 → 12400
```

**계속 동기화:**

```
Master에서 계속 쓰기:
  12:00:14.000 - INCR counter
  12:00:15.000 - DEL temp:key
  12:00:16.000 - HSET order:123 status "배달중"

각 명령이 즉시 Slave로 전송됨:
  INCR counter → Slave 실행
  DEL temp:key → Slave 실행
  HSET order:123 status "배달중" → Slave 실행
```

**이렇게 계속 실시간으로 동기화된다.**

---

### 4단계: 연결 끊김 및 재연결

**네트워크 장애 발생:**

```
Master ─ X ─ Slave (연결 끊김)
```

네트워크 케이블이 빠지거나, 방화벽 문제, 라우터 장애 등으로 연결이 끊길 수 있다.

**Master는 계속 동작:**

```
Master (혼자):
  클라이언트 요청 계속 처리
  SET user:5 "수진"
  INCR counter
  DEL old:key
  HSET order:456 status "완료"
  
  Slave로 보내려고 하지만 연결 끊김
  → Replication Backlog에 저장
```

---

**Replication Backlog가 뭔가?**

**Replication Backlog는 원형 버퍼(Circular Buffer)다.**

```
┌─────────────────────────────┐
│ Replication Backlog (1MB)   │
│                             │
│ ┌─────────────────────────┐ │
│ │ SET user:5 "수진"       │ │ ← 명령들이 계속 쌓임
│ │ INCR counter            │ │
│ │ DEL old:key             │ │
│ │ HSET order:456 ...      │ │
│ │ SET user:6 "민지"       │ │
│ │ ...                     │ │
│ └─────────────────────────┘ │
└─────────────────────────────┘
```

**원형 버퍼의 특징:**

- 고정 크기 (기본 1MB)
- 새로운 데이터가 들어오면 오래된 데이터를 덮어씀
- 최근 명령들만 유지

**왜 필요한가?**

Slave가 잠깐 연결이 끊겼다가 다시 연결되면, 그 사이에 뭐가 바뀌었는지 알려줘야 한다. 그때 이 Backlog를 사용한다.

---

**연결 복구:**

```
Master ─ ✓ ─ Slave (연결 복구됨)
```

**Slave의 요청:**

```
Slave → Master:
  "PSYNC 8371b4fb1155b71f4a04d3e1bc3e18c4a990aeeb 12400"
  
  의미:
  - Master ID: 8371...
  - 마지막 Offset: 12400
  - "12400 지점부터 주세요"
```

**Master의 판단:**

```
Master:
  1. Backlog 확인
  2. Offset 12400이 Backlog에 있나?
  
  경우 1: 있음
    "CONTINUE" 응답
    12400 이후 명령만 전송:
      SET user:5 "수진"
      INCR counter
      DEL old:key
      HSET order:456 ...
    
  경우 2: 없음 (Backlog가 작아서 덮어써짐)
    "FULLRESYNC ..." 응답
    RDB 파일 다시 생성 & 전송
```

**경우 1 (Partial Sync 성공):**

```
Slave:
  Master로부터 명령 수신:
    SET user:5 "수진"
    INCR counter
    DEL old:key
    HSET order:456 ...
  
  명령 실행:
    각 명령을 순서대로 실행
  
  동기화 완료!
```

빠르게 따라잡을 수 있다. RDB 파일 전송 없이 명령만 실행하면 되니까.

---

**경우 2 (Partial Sync 실패):**

```
연결이 1시간 동안 끊겼다고 가정:
  
  Backlog 크기: 1MB
  1시간 동안 쓰기: 100MB 발생
  
  → Backlog는 최근 1MB만 유지
  → Offset 12400은 이미 덮어써짐
  
  Master:
    "미안, 12400은 이미 없어. Full Sync 해야 해"
    FULLRESYNC 응답
    
  Slave:
    RDB 파일 받기
    처음부터 다시 동기화
```

느리다. 하지만 어쩔 수 없다.

---

**Backlog 크기 조정:**

```conf
# redis.conf
repl-backlog-size 10mb  # 10MB로 증가
```

크게 설정하면:
- 장점: 재연결 시 Partial Sync 성공 확률 높아진다
- 단점: 메모리 더 사용한다

**권장:**
- 짧은 장애 (수 초~수 분): 1MB로 충분
- 긴 장애 가능성 (수십 분): 10MB~100MB

## 비동기 복제와 데이터 일관성

**Master와 Slave는 완벽하게 동시에 같지 않다.**

이게 중요한 부분이다.

### Replication Lag (복제 지연)

**복제는 비동기로 일어난다.**

```
시각 12:00:00.000
  클라이언트 → Master: SET user:1 "철수"

시각 12:00:00.000 (즉시)
  Master: 명령 실행 완료
  Master → 클라이언트: "OK"

시각 12:00:00.001 (1ms 후)
  Master → Slave: "SET user:1 철수" 전송 시작
  (네트워크 통과 중)

시각 12:00:00.010 (10ms 후)
  Slave: 명령 수신
  Slave: 명령 실행
  Slave: 완료
```

**중요한 점:**

Master는 Slave의 응답을 기다리지 않고 바로 클라이언트에게 "OK"를 보낸다.

**이 사이 10ms 동안:**

```
Master: user:1 = "철수"
Slave: user:1 = (아직 업데이트 안 됨, 이전 값 또는 없음)
```

Master와 Slave가 다르다!

---

**왜 비동기인가?**

**만약 동기 복제라면:**

```
클라이언트 → Master: SET user:1 "철수"

Master:
  1. 명령 실행
  2. Slave로 전송
  3. Slave의 "OK" 대기
  4. Slave로부터 "OK" 수신
  5. 클라이언트에게 "OK" 전송
```

Slave가 3대라면 3대 모두의 응답을 기다려야 한다.

```
Master → Slave 1: 전송 & 대기 (10ms)
Master → Slave 2: 전송 & 대기 (10ms)
Master → Slave 3: 전송 & 대기 (10ms)

총 지연: 30ms
```

**문제:**
- 모든 쓰기가 30ms씩 느려진다
- Slave 중 하나라도 느리면 전체가 느려진다
- Slave 중 하나라도 죽으면 쓰기 불가능

**그래서 Redis는 비동기 복제를 사용한다:**

```
Master:
  명령 실행 → 즉시 "OK"
  Slave 전송은 백그라운드에서
  
빠르다!
```

---

### 실무에서의 문제

**시나리오: 게시글 작성**

사용자가 게시글을 작성한다.

```
1. 사용자: "게시글 작성" 버튼 클릭

2. 백엔드: Master에 저장
   masterRedis.set("post:123", "내용...")
   응답: "OK"

3. 리다이렉트: "게시글 목록" 페이지로 이동

4. 게시글 목록 조회: Slave에서 읽기
   List<String> posts = slaveRedis.zrange("posts:recent", 0, 9);
   
5. 화면 렌더링
```

**문제:**

2번에서 Master에 저장했다. 하지만 4번에서 Slave로 조회할 때 아직 복제가 안 되었을 수 있다.

```
시각 12:00:00.000
  Master 저장 완료
  
시각 12:00:00.050 (50ms 후)
  리다이렉트 완료, 목록 페이지 요청
  
시각 12:00:00.060 (60ms 후)
  Slave 조회
  
시각 12:00:00.070 (70ms 후)
  Slave 복제 완료 (늦음!)
```

**사용자가 보는 것:**

```
"게시글을 작성했는데 목록에 안 보여요!"
```

---

**해결책 1: 쓰기 직후에는 Master에서 읽기**

```java
public PostResponse createPost(String content) {
    String postId = UUID.randomUUID().toString();
    
    // Master에 저장
    masterRedis.set("post:" + postId, content);
    masterRedis.zadd("posts:recent", System.currentTimeMillis(), postId);
    
    // 바로 읽어야 하면 Master에서 읽기
    String savedPost = masterRedis.get("post:" + postId);
    
    return new PostResponse(postId, savedPost);
}
```

Master에서 읽으면 방금 쓴 데이터를 확실히 볼 수 있다.

---

**해결책 2: 잠시 대기**

```java
public void createPost(String content) {
    // Master에 저장
    masterRedis.set("post:" + postId, content);
    
    // 복제 시간 대기
    Thread.sleep(100);  // 100ms
    
    // 이제 Slave에서 읽기
    // (대부분 복제 완료됨)
}
```

하지만 사용자 경험이 안 좋다. 100ms 느려지니까.

---

**해결책 3: 낙관적 UI + 비동기 갱신**

```javascript
// 프론트엔드
function createPost(content) {
  // 1. 화면에 즉시 표시 (낙관적 UI)
  displayPost({id: 'temp', content: content});
  
  // 2. 백엔드에 저장
  api.post('/posts', {content})
    .then(response => {
      // 3. 실제 ID로 교체
      replacePost('temp', response.id);
    });
}
```

사용자는 즉시 본인 글을 보고, 백그라운드에서 저장된다.

---

**해결책 4: Eventually Consistent 받아들이기**

```
"일시적으로 안 보일 수 있지만, 곧 보입니다"

대부분의 경우 100ms 이내에 복제되므로
사용자가 새로고침하면 보인다
```

Facebook, Twitter 같은 대규모 서비스도 이 방식을 쓴다.

## Master 장애 시나리오

**Master가 다운되면 어떻게 될까?**

### 장애 발생

```
        Master
         ↓ (죽음)
       /  |  \
      /   |   \
  Slave Slave Slave
   (1)   (2)   (3)
```

**결과:**

```
읽기:
  Slave들이 살아있으므로 계속 가능
  사용자는 데이터 조회 가능

쓰기:
  Master가 죽었으므로 불가능!
  SET, INCR 등 모든 쓰기 실패
```

**서비스 영향:**

```
읽기 전용 모드:
  - 배달 현황 조회: 가능
  - 주문 접수: 불가능
  - 위치 업데이트: 불가능
```

---

### 수동 Failover (장애 전환)

**사람이 직접 개입해서 Slave를 Master로 승격시켜야 한다.**

**절차:**

```
1. 상황 파악
   모니터링 시스템에서 알림
   "Master가 응답하지 않습니다!"

2. Master 상태 확인
   정말 죽었는지 확인
   (네트워크 일시 장애일 수도 있음)

3. Slave 중 하나 선택
   가장 최신 데이터를 가진 Slave
   (Replication Offset이 가장 큰 것)
   
   예: Slave 1 선택

4. Slave 1을 Master로 승격
   redis-cli -h 192.168.1.101
   REPLICAOF NO ONE
   
   → Slave 1이 독립적인 Master가 됨

5. 다른 Slave들을 새 Master로 연결
   Slave 2:
     redis-cli -h 192.168.1.102
     REPLICAOF 192.168.1.101 6379
   
   Slave 3:
     redis-cli -h 192.168.1.103
     REPLICAOF 192.168.1.101 6379

6. 애플리케이션 설정 변경
   application.yml:
     redis:
       host: 192.168.1.100  → 192.168.1.101

7. 애플리케이션 재배포
   모든 서버 재시작
```

**문제점:**

```
- 수동 작업 (사람이 직접)
- 시간 소요 (최소 수 분)
- 실수 가능성
- 새벽에 장애 나면?
- 담당자가 휴가 중이면?
```

**해결책:**

Redis Sentinel을 사용하면 이 모든 과정이 자동화된다.

## 정리

### Master-Slave란?

```
Master:
  - 원본 데이터 관리
  - 쓰기 작업 처리
  - Slave로 복제 전송
  - 읽기도 가능하지만 보통 안 씀

Slave:
  - 복사본 데이터 보유
  - 읽기 전용 (기본)
  - Master로부터 데이터 수신
  - 쓰기 불가능
```

### 왜 사용하나?

```
1. 읽기 성능 향상
   Slave 추가로 읽기 분산

2. 장애 대응
   Master 죽어도 Slave 있음

3. 백업
   Slave에서 RDB 저장
```

### 복제 방식

```
처음: Full Sync (RDB 전송)
이후: Partial Sync (명령어 전송)
재연결: Backlog로 복구
```

### 주의사항

```
비동기 복제 (지연 있음)
Master 죽으면 쓰기 불가
수동 Failover 필요
Slave는 읽기 전용 유지
```
