# Redis Sentinel

## Sentinel이 뭔가?

**Sentinel = 감시자, 파수꾼이란 뜻이다**

Redis Sentinel은 **Redis 서버들을 감시하고 자동으로 장애를 처리해주는 시스템**이다.

Master-Slave 구조에서 Master가 죽으면 수동으로 Slave를 Master로 승격시켜야 했다. 이 과정을 **자동으로** 해주는 게 Sentinel이다.

---

## 왜 Sentinel이 필요한가?

Master-Slave만으로는 부족한 이유를 먼저 이해해야 한다.

### Master-Slave의 문제점

**상황:**
새벽 3시, Master 서버가 갑자기 다운된다.

```
        Master (죽음!)
         ↓ 
       /  |  \
      /   |   \
  Slave Slave Slave
   (1)   (2)   (3)
```

**어떻게 될까?**

```
1. 읽기: Slave들이 살아있어서 가능
2. 쓰기: Master가 없어서 불가능!
   - 주문 접수 안 됨
   - 게시글 업로드 안 됨
   - 모든 쓰기 작업 실패
```

**서비스는 사실상 중단 상태다.**

---

### 수동 복구의 문제

**담당자가 해야 할 일:**

```
1. 모니터링 알림 받음

2. Master 상태 확인
   정말 죽었는지 확인
   (네트워크 일시 장애일 수도 있음)

3. Slave 중 하나 선택
   INFO replication으로 offset 확인
   가장 최신 데이터를 가진 Slave 찾기

4. 선택한 Slave를 Master로 승격
   redis-cli -h 192.168.1.101
   REPLICAOF NO ONE

5. 다른 Slave들을 새 Master로 연결
   Slave 2: REPLICAOF 192.168.1.101 6379
   Slave 3: REPLICAOF 192.168.1.101 6379

6. 애플리케이션 설정 파일 수정
   redis.host=192.168.1.100 → 192.168.1.101

7. 모든 애플리케이션 서버 재배포
   배포 스크립트 실행
   서버들 재시작 대기

8. 서비스 정상 동작 확인
```

**문제점:**

```
사람이 직접 개입 필요
   - 잠자고 있을 수도
   - 휴가 중일 수도
   - 술 마시고 있을 수도

시간이 오래 걸림
   - 최소 10분 이상
   - 그 사이 쓰기 불가능
   - 매출 손실

실수 가능성
   - offset 낮은 Slave 선택
   - 설정 파일 잘못 수정
   - 새벽이라 정신 없음

복구 중 추가 장애 가능
   - Slave도 죽으면?
   - 네트워크 또 끊기면?
```

---

### Sentinel로 해결하기

**Sentinel이 자동으로 처리한다:**

```
Master 다운
  ↓
Sentinel이 즉시 감지 (1초 이내)
  ↓
다른 Sentinel들과 합의
  ↓
자동으로 Slave 1을 Master로 승격
  ↓
다른 Slave들을 새 Master로 연결
  ↓
애플리케이션에게 새 Master 주소 알림
  ↓
서비스 계속 가능
```

담당자는 확인만 하면 된다

```
[Redis Sentinel] 03:24 AM
Master 192.168.1.100:6379 다운 감지
Slave 192.168.1.101:6379를 새 Master로 승격
Failover 완료
```

---

## Sentinel의 3가지 핵심 기능

### 1. Monitoring (모니터링)

**계속 감시한다:**

```
Sentinel이 하는 일:
  매 초마다 Master에게 PING 전송
  매 초마다 모든 Slave에게 PING 전송
  매 초마다 다른 Sentinel들에게 PING 전송
  
"살아있나?" 계속 확인
```

**PING/PONG:**

```
Sentinel → Master: PING
Master → Sentinel: PONG (살아있어요)

Sentinel → Master: PING
Master → Sentinel: ... (응답 없음)

Sentinel → Master: PING
Master → Sentinel: ... (또 응답 없음)

Sentinel: "Master 죽은 것 같은데?"
```

---

### 2. Notification (알림)

**문제가 생기면 알려준다:**

```
Master 다운 감지
  ↓
Slack, 이메일, SMS 등으로 알림
  ↓
담당자가 상황 파악 가능
```

알림 예시:
```
[Redis Sentinel Alert]
Master mymaster (192.168.1.100:6379) is down
Slave 192.168.1.101:6379 promoted to master
All slaves reconfigured
Failover completed in 12 seconds
```

---

### 3. Automatic Failover (자동 장애 전환)

**자동으로 Master를 바꾼다:**

```
Master 죽음 확인
  ↓
Sentinel들끼리 투표
  ↓
과반수 찬성
  ↓
Slave 중 하나를 새 Master로 선택
  ↓
선택된 Slave: REPLICAOF NO ONE
  ↓
다른 Slave들: REPLICAOF 새_Master
  ↓
애플리케이션에게 새 Master 알림
  ↓
완료!
```

REPLICAOF는 누구를 Master로 따를 것인가를 설정하는 명령어가

NO ONE은 누구도 따르지 않겠다는 설정이며, 해당 Slave가 Master로 전환된다는 명령어다

다른 Slave들은 새_Master를 따른다고 설정한다

**핵심: 사람 개입 없이 자동으로!**

## Sentinel 구조

### 기본 구조

Sentinel도 Redis 서버다. 다만 데이터 저장 안 하고 감시만 한다.

```
     Sentinel 1        Sentinel 2        Sentinel 3
         ↓                 ↓                 ↓
         └─────────────────┴─────────────────┘
                           ↓
                  계속 감시 & 서로 통신
                           ↓
                 ┌─────────┴─────────┐
                 ↓                   ↓
              Master              Slave 1, 2, 3
```

**왜 Sentinel을 여러 개 쓸까?**

Sentinel 1대만 있으면:
- Sentinel이 죽으면? 감시 중단
- Sentinel이 잘못 판단하면? 오작동

그래서 **최소 3대**를 둔다. 서로 확인하고 투표한다.

---

### 권장 구성

**최소 구성: Sentinel 3대**

```
Sentinel 1 (서버 A)
Sentinel 2 (서버 B)
Sentinel 3 (서버 C)
  ↓
Master (서버 D)
Slave 1 (서버 E)
Slave 2 (서버 F)
```

**왜 3대?**

```
2대면:
  Sentinel 1: "Master 죽었어"
  Sentinel 2: "아니야, 네가 네트워크 끊긴 거야"
  → 투표 결과 1:1, 결론 못 냄

3대면:
  Sentinel 1: "Master 죽었어"
  Sentinel 2: "맞아, 죽었어"
  Sentinel 3: "응답 없음"
  → 투표 결과 2:1, 과반수 찬성, Failover 진행
```

---

### Sentinel과 Redis 서버를 분리해야 하나?

**분리하는 게 좋다.**

**나쁜 예: Sentinel 2개가 같은 서버에 설치**

```
서버 A:
  Master
  Sentinel 1
  Sentinel 2

서버 A 전체 다운
  ↓
Master도 죽음
Sentinel 1, 2도 죽음
  ↓
Sentinel 3만 남음
```

Quorum을 2로 설정했을 경우 Sentinel 투표 수가 1개라서 과반수를 못채운다

**좋은 예: 다른 서버에 설치**

```
서버 A: Master
서버 B: Slave 1
서버 C: Slave 2
서버 D: Sentinel 1
서버 E: Sentinel 2
서버 F: Sentinel 3
```

물론, 이렇게 하더라도 Sentinel 1, 2 모두 다운되면 역시나 과반수를 못채우는건 마찬가지지만

같은 서버에 두는 것보다는 훨씬 안전하다

---

## Quorum (쿼럼)이 뭔가?

**Quorum = 의사결정에 필요한 최소 인원**

Sentinel이 "Master 죽었다"고 판단하려면 **최소 몇 명의 Sentinel이 동의**해야 하는지 정하는 숫자다.

### Quorum 설정

**sentinel.conf:**

```conf
sentinel monitor mymaster 192.168.1.100 6379 2
                                              ↑
                                          quorum = 2
```

이 뜻은:
```
"mymaster라는 Master를 감시하는데,
최소 2명의 Sentinel이 '죽었다'고 동의해야
실제로 죽은 걸로 인정한다"
```

---

### Quorum 예시

**Sentinel 3대, Quorum 2:**

```
Sentinel 1: "Master에 PING 보냈는데 응답 없음"
  → 혼자만 알고 있음 (1/2)
  → 아직 Failover 안 함

Sentinel 2: "나도 Master 응답 없음"
  → 2명이 동의함 (2/2)
  → Quorum 충족!
  → Failover 시작

Sentinel 3: "어? 나는 응답 받았는데..."
  → 소수 의견, 무시됨
```

---

### Quorum은 어떻게 정하나?

**공식: Quorum = (Sentinel 총 개수 / 2) + 1**

```
Sentinel 3대: Quorum = 2
Sentinel 5대: Quorum = 3
Sentinel 7대: Quorum = 4
```

**왜 과반수가 중요할까?**

과반수보다 적으면 **네트워크 분리(Split Brain)** 문제가 생긴다.

네트워크 분리란 서버들 사이의 네트워크 연결이 끊어져서 두 그룹으로 나뉘는걸 말하는데

이때 각 그룹에서 마스터를 선출해서 생기는 문제다

```
데이터센터 X          데이터센터 Y
┌─────────┐           ┌─────────┐
│ Master  │───────────│ Slave 1 │
│Sentinel1│           │Sentinel2│
└─────────┘           │Sentinel3│
                      └─────────┘

네트워크 연결: 정상
서로 통신 가능

데이터센터 X          데이터센터 Y
┌─────────┐           ┌─────────┐
│ Master  │  X X X X  │ Slave 1 │
│Sentinel1│  (단절)   │Sentinel2│
└─────────┘           │Sentinel3│
                      └─────────┘

X와 Y 사이 연결 끊김
각자는 정상 동작 중
```

**Split Brain 시나리오:**

```
데이터센터 X:
  서버 A: Master + Sentinel 1

데이터센터 Y:
  서버 B: Slave 1 + Sentinel 2
  서버 C: Slave 2 + Sentinel 3

총 Sentinel 3개
Quorum 2


네트워크 단절이 발생하면:

서버 A는 이미 Master가 존재하기 때문에 Failover는 발생하지 않음

서버 B는 Master와 통신이 불가능해져서 Sentienl 2, 3 중 리더를 선출함


네트워크 통신 복구:

네트워크 통신이 복구되고 쓰기 작업을 받았는데

Master가 2개라서 다시 투표 진행

둘 중 하나는 Master로, 하나는 Slave로 강등되는데

이때, Slave로 강등되면 해당 데이터는 유실된다
```

Failover란 장애 전환을 말하는데 여기서는 Master를 교체하는 작업을 말한다

## Sentinel 동작 원리 (자세히)

### 1. 정상 상태: 계속 감시

**매 초마다:**

```
Sentinel 1:
  → Master에게 PING
  → Slave 1에게 PING
  → Slave 2에게 PING
  → Sentinel 2에게 PING
  → Sentinel 3에게 PING

Sentinel 2:
  → Master에게 PING
  → Slave 1에게 PING
  → Slave 2에게 PING
  → Sentinel 1에게 PING
  → Sentinel 3에게 PING

Sentinel 3:
  → Master에게 PING
  → Slave 1에게 PING
  → Slave 2에게 PING
  → Sentinel 1에게 PING
  → Sentinel 2에게 PING
```

**모두 PONG 응답:**

```
상태: 정상
아무 일도 안 일어남
계속 감시만
```

---

### 2. Master 다운 감지 예시

**시각 03:24:00**

```
Sentinel 1 → Master: PING
Master: ... (응답 없음)

Sentinel 1 내부:
  "어? 응답 없네"
  "한 번 더 보내보자"
```

**시각 03:24:01**

```
Sentinel 1 → Master: PING
Master: ... (또 응답 없음)

Sentinel 1 내부:
  "2번 연속 응답 없음"
  "일시적인 네트워크 문제일 수도"
  "조금 더 기다려보자"
```

**시각 03:24:02 ~ 03:24:30**

```
계속 PING 보내지만 응답 없음

30초 경과 (down-after-milliseconds 기본값)

Sentinel 1:
  "30초 동안 응답 없음"
  "이건 진짜 죽은 것 같다"
  "S_DOWN(Subjectively Down) 상태로 표시"
```

**S_DOWN = 주관적 다운**

Sentinel 1 혼자만 "죽었다"고 판단한 상태다. 아직 확실하지 않다.

---

### 3. 다른 Sentinel에게 물어봄

**Sentinel 1:**

```
"Sentinel 2, 3, 너희도 Master 응답 없어?"
```

**Sentinel 2:**

```
Sentinel 2 → Master: PING
Master: ... (응답 없음)

"나도 응답 없어!"
```

**Sentinel 3:**

```
Sentinel 3 → Master: PING
Master: ... (응답 없음)

"나도 없어!"
```

**결과:**

```
Sentinel 1: Master 죽음 (1표)
Sentinel 2: Master 죽음 (2표)
Sentinel 3: Master 죽음 (3표)

총 3표 → Quorum 2 충족!
```

**O_DOWN(Objectively Down) 상태로 변경**

객관적으로 죽었다고 판단. 이제 Failover 시작.

---

### 4. Leader Sentinel 선출

**Raft 알고리즘 기반**

Sentinel은 Raft 합의 알고리즘을 사용해서 Leader를 선출한다

Raft 합의 알고리즘은 여러 서버가 하나의 결정에 합의하는 방법을 말한다

선출 과정 (상세)

1. Epoch 개념

```
Epoch = 선거 라운드 번호

초기: current_epoch = 0

Master 다운 감지:
  각 Sentinel이 current_epoch를 1 증가
  current_epoch = 1
```

Epoch는 논리적 시간으로 각 선거마다 증가한다

<br>

2. 투표 요청

```
Sentinel 1이 O_DOWN 감지:
  1. current_epoch 증가: 0 → 1
  2. 자신의 leader_epoch를 1로 설정
  3. 다른 Sentinel들에게 투표 요청 전송
  
메시지:
  "SENTINEL is-master-down-by-addr <ip> <port> <current_epoch> <runid>"
  
  예:
  "SENTINEL is-master-down-by-addr 192.168.1.100 6379 1 8f7e..."
  
  의미:
  - 192.168.1.100:6379 Master 죽었음
  - 현재 epoch: 1
  - 나(8f7e...)를 Leader로 뽑아줘
```

<br>

3. 투표 규칙

**각 Sentinel은 epoch당 1표만**


**투표 승인**

```
Sentinel 2가 받은 투표 요청:
  
  확인 사항:
  1. 요청의 epoch가 내 current_epoch보다 크거나 같은가?
     - 작으면 무시 (오래된 요청)
     - epoch는 n대 대선이라고 생각하면 된다
     - 22대 대통령 선거를 이미 치뤘는데 요청으로 22가 오면 이미 완료된 투표라 무시한다
  
  2. 이번 epoch에 이미 투표했나?
     - 했으면 거절
     - 안 했으면 승인 가능
  
  3. Master가 정말 다운됐나?
     - 자신도 O_DOWN 판정했는지 확인
     - 안 했으면 거절
```

**투표 거절**

```
Sentinel 2:
  모든 조건 만족
  → Sentinel 1에게 투표
  → leader_epoch = 1, voted_leader = Sentinel 1
  
응답:
  "1 8f7e..."
  
  의미:
  - epoch 1에서
  - 8f7e...(Sentinel 1)에게 투표함
```

<br>

4. 과반수 획득

```
Sentinel 3대, Quorum 2:

Sentinel 1 투표 현황:
  자기 자신: 1표
  Sentinel 2: 1표 (승인)
  Sentinel 3: 1표 (승인)
  
총 3표 (과반수 = 2)

Sentinel 1:
  "과반수 획득!"
  "나는 epoch 1의 Leader다"
  leader = myself
  leader_epoch = 1
```

<br>

5. 동시 요청 처리

**경쟁 상황**

```
Sentinel 1과 Sentinel 2가 거의 동시에 O_DOWN 감지:

시각 03:24:30.000
  Sentinel 1: epoch 증가 (0→1), 투표 요청 전송
  
시각 03:24:30.002
  Sentinel 2: epoch 증가 (0→1), 투표 요청 전송
```

**Sentinel 3의 처리**

```
시각 03:24:30.001
  Sentinel 1 요청 수신
  → epoch 1, 아직 투표 안 함
  → Sentinel 1에게 투표
  → voted_leader = Sentinel 1

시각 03:24:30.003
  Sentinel 2 요청 수신
  → epoch 1, 이미 Sentinel 1에게 투표함
  → 거절
```

**결과**

```
Sentinel 1:
  자기: 1표
  Sentinel 3: 1표
  총 2표 (과반수)
  → Leader!

Sentinel 2:
  자기: 1표
  총 1표 (과반수 못 채움)
  → Leader 실패
```

**먼저 도착한 요청이 유리하다**

<br>

6. 분할 투표

**경쟁이 심한 경우**

```
Sentinel 5대, Quorum 3:

epoch 1 투표:
  Sentinel 1: 2표 (본인투표 + Sentinel 4)
  Sentinel 2: 2표 (본인투표 + Sentinel 5)
  Sentinel 3: 1표 (본인투표)
  
→ 아무도 과반수(3표) 못 채움
```

**재선거**

```
일정 시간 대기 (1초 정도)
  ↓
epoch 증가: 1 → 2
  ↓
다시 투표 시작
  ↓
이번엔 Sentinel 1이 먼저 요청
  ↓
Sentinel 1: 3표
  ↓
Leader
```

**투표 요청 메시지 상세**

```
명령 구조:
  SENTINEL is-master-down-by-addr <ip> <port> <current-epoch> <runid>

예시:
  SENTINEL is-master-down-by-addr 192.168.1.100 6379 5 abc123...

파라미터:
  ip: Master IP
  port: Master 포트
  current-epoch: 현재 epoch (선거 라운드)
  runid: 투표 요청하는 Sentinel의 고유 ID
    - "*" 이면 단순 확인 (투표 요청 아님)
    - ID 있으면 투표 요청
```

**응답**

```
배열 응답:
  1) <is_down>: 1 또는 0
     1 = Master 다운 동의
     0 = Master 살아있다고 봄
  
  2) <leader>: Leader로 뽑힌 Sentinel의 runid -> runid는 Redis/Sentinel 인스턴스의 고유 ID
     "*" = 아직 Leader 없음
     "abc123..." = 해당 Sentinel이 Leader
  
  3) <vote-epoch>: 투표한 epoch

예:
  1) 1
  2) "abc123..."
  3) 5
  
  의미:
  - Master 다운 동의 (1)
  - epoch 5에서
  - abc123... Sentinel에게 투표함
```

**왜 runid를 쓸까?**

IP:Port로는 부족하기 때문

```
문제 상황:
  Sentinel 1 (192.168.1.201) 재시작
  같은 IP:Port로 다시 시작
  
  다른 Sentinel들:
  "192.168.1.201은 아직 내가 뽑은 Leader인데?"
  
  하지만 실제로는:
  다른 인스턴스 (재시작됨)
```

runid로 구분

### 5. 애플리케이션 자동 연결

**Sentinel은 애플리케이션에게 새 Master를 알려준다.**

애플리케이션이 Sentinel을 통해 Redis에 연결했다면:

```java
// Spring Boot 설정
spring:
  redis:
    sentinel:
      master: mymaster
      nodes:
        - 192.168.1.201:26379  # Sentinel 1
        - 192.168.1.202:26379  # Sentinel 2
        - 192.168.1.203:26379  # Sentinel 3
```

이렇게 설정하면:

```
1. 애플리케이션이 Sentinel에게 물어봄
   "mymaster Master 주소가 뭐야?"

2. Sentinel 응답
   "192.168.1.101:6379야"

3. 애플리케이션이 해당 주소로 연결

4. Failover 발생
   Master 주소 변경: 192.168.1.101 → 192.168.1.102

5. 애플리케이션이 다시 물어봄
   "mymaster Master 주소가 뭐야?"

6. Sentinel 응답
   "192.168.1.102:6379야" (자동으로 바뀐 주소)

7. 애플리케이션이 새 Master로 연결
```

**사람이 설정 파일 수정하거나 재배포할 필요 없다**

### 6. 죽었던 Master가 다시 살아나면?

**시각 04:00:00**

옛날 Master가 재부팅되어 다시 살아난다.

```
옛날 Master (192.168.1.100):
  "나 다시 살아났어!"
  "내가 Master였는데..."
```

**Sentinel이 감지:**

```
Sentinel들:
  "어? 192.168.1.100이 살아났네"
  "근데 이제 Master는 192.168.1.101이야"
  "얘를 Slave로 만들어야겠다"

Sentinel 2 → 옛날 Master: REPLICAOF 192.168.1.101 6379

옛날 Master:
  "아... 알겠어..."
  역할 변경: Master → Slave
```

**최종 구조:**

```
        새 Master (192.168.1.101)
       /         |              \
   Slave 2   Slave 3   옛날 Master (Slave로 전환)
```

**자동으로 Slave로 편입된다!**

## Sentinel 주요 설정 상세

### down-after-milliseconds

**Master가 죽었다고 판단하는 시간**

```conf
sentinel down-after-milliseconds mymaster 30000
# 30초 (30,000 밀리초)
```

**동작:**

```
PING 응답 없음
  ↓
30초 대기
  ↓
여전히 응답 없음
  ↓
S_DOWN (주관적 다운) 판정
```

**너무 짧으면:**
```
일시적 네트워크 지연에도 Failover
→ 불필요한 Failover 빈번

예: 5초 설정
  네트워크 5초 지연
  → Failover 발동
  → 곧 네트워크 복구
  → 괜히 Master 바뀜
```

**너무 길면:**
```
실제 장애 시 복구 느림

예: 5분 설정
  Master 진짜 죽음
  → 5분 대기
  → 5분 동안 쓰기 불가
```

**권장:**
```
30초 ~ 60초
```

---

### failover-timeout

**Failover가 완료되어야 하는 시간**

```conf
sentinel failover-timeout mymaster 180000
# 3분 (180,000 밀리초)
```

**용도:**

1. **Failover 전체 시간 제한**
   ```
   Failover 시작
     ↓
   3분 경과
     ↓
   아직 완료 안 됨
     ↓
   실패 처리
   ```

2. **Slave 응답 대기 시간**
   ```
   Sentinel → Slave: "Master 돼"
     ↓
   Slave 응답 대기
     ↓
   timeout 시간 초과
     ↓
   다른 Slave 선택
   ```

3. **재시도 간격**
   ```
   Failover 실패
     ↓
   timeout 시간의 2배 대기
     ↓
   다시 시도
   ```

**권장:**
```
3분 ~ 5분
```

---

### parallel-syncs

**동시에 복제할 Slave 수**

```conf
sentinel parallel-syncs mymaster 1
```

**의미:**

Failover 후 새 Master에 Slave들이 연결될 때:

```
parallel-syncs 1:
  Slave 1 → 새 Master 복제 (Full Sync)
  완료
  Slave 2 → 새 Master 복제
  완료
  Slave 3 → 새 Master 복제
  완료
  
  → 한 번에 1대씩

parallel-syncs 2:
  Slave 1, 2 → 새 Master 복제 (동시)
  완료
  Slave 3 → 새 Master 복제
  완료
  
  → 한 번에 2대씩

parallel-syncs 10:
  Slave 1, 2, 3 → 새 Master 복제 (모두 동시)
  완료
  
  → 전부 동시
```

**parallel-syncs를 왜 제한할까?**

```
Full Sync는 부담이 크다:
  - 새 Master가 RDB 생성
  - 네트워크로 전송
  - CPU, 메모리, 네트워크 사용량 증가

여러 Slave가 동시에 Full Sync 하면:
  → 새 Master 과부하
  → 성능 저하
  → 최악의 경우 다운
```

**권장:**
```
1 또는 2
```

## Sentinel의 한계

### 1. 쓰기 성능은 여전히 1대

```
Master 1대만 쓰기 가능
  ↓
Slave 아무리 많아도 쓰기 성능 동일
  ↓
쓰기 병목 해결 안 됨
```

**해결책:**
Redis Cluster

---

### 2. 데이터 유실 가능성

**비동기 복제의 한계:**

```
시각 03:24:00.000
  클라이언트 → Master: SET key "value"
  Master: 저장 완료, "OK" 응답

시각 03:24:00.001
  Master → Slave로 복제 시작
  
시각 03:24:00.002
  Master 다운! (복제 안 됨)
  
시각 03:24:45
  Slave를 Master로 승격
  
결과:
  SET key "value" 유실됨
```

**최소화 방법:**

```conf
# Master에 최소 1개의 Slave가 연결되어야 쓰기 허용
min-replicas-to-write 1

# Slave의 복제 지연이 10초 이하여야 쓰기 허용
min-replicas-max-lag 10
```

완벽하진 않지만 유실 확률을 줄인다.

# Redis Cluster

## Cluster가 뭔가?

**Cluster = 여러 대의 Redis 서버를 하나처럼 사용하는 시스템**

Master-Slave는 Master가 1대뿐이었다. 모든 쓰기가 그 1대로 몰렸다.

Cluster는 **Master를 여러 대** 둔다. 데이터를 나눠서 저장하고, 쓰기도 분산한다.

**비유:**

```
Master-Slave:
  음식점 1개 (Master)
  배달원 3명 (Slave)
  
  문제: 주문은 음식점 1개에서만 처리
       → 음식점 과부하

Cluster:
  음식점 3개 (Master 3개)
  각 음식점마다 배달원 1명 (각 Master마다 Slave 1개)
  
  해결: 주문을 3개 음식점이 나눠서 처리
       → 3배 빠름
```

---

## 왜 Cluster가 필요한가?

Master-Slave + Sentinel의 한계를 먼저 이해해야 한다.

### Master-Slave + Sentinel의 한계

**상황:**
대규모 서비스를 운영 중이다.

```
구성:
  Master 1대 (쓰기 처리)
  Slave 10대 (읽기 처리)
  Sentinel 3대 (감시)
```

**읽기는 괜찮다:**
```
읽기 요청: 초당 100만 건
Slave 10대가 나눠서 처리
→ 각 Slave당 10만 건
→ 처리 가능
```

**쓰기가 문제다:**
```
쓰기 요청: 초당 20만 건
Master 1대가 혼자 처리
→ Master 한계: 10만 QPS
→ 처리 불가능
→ 타임아웃, 장애
```

**Slave를 아무리 늘려도 쓰기 성능은 안 올라간다**

---

### 메모리 한계

```
데이터 크기: 100GB
Master 메모리: 64GB

→ 데이터가 안 들어감!
```

**메모리를 늘리면?**
```
서버 1대 메모리: 최대 512GB 정도
데이터가 1TB면?
→ 한 대에 안 들어감
```

---

### 해결책: Cluster

**여러 대의 Master를 둔다:**

```
Master 3대:
  Master 1: 데이터 1/3
  Master 2: 데이터 1/3
  Master 3: 데이터 1/3
```

**쓰기 분산:**
```
쓰기 요청 30만 건:
  Master 1: 10만 건 처리
  Master 2: 10만 건 처리
  Master 3: 10만 건 처리
  
→ 3배 성능
```

**메모리 분산:**
```
데이터 300GB:
  Master 1: 100GB
  Master 2: 100GB
  Master 3: 100GB
  
각 서버 메모리: 128GB
→ 여유 있음
```

---

## Cluster vs Master-Slave 비교

### Master-Slave + Sentinel

```
        Master (쓰기)
       /   |   \
   Slave Slave Slave (읽기)

장점:
  - 읽기 성능 확장 가능
  - 자동 Failover (Sentinel)

단점:
  - 쓰기 성능 1대 한계
  - 메모리 1대 한계
  - 데이터 분산 불가
```

---

### Cluster

```
Master 1 - Slave 1
Master 2 - Slave 2
Master 3 - Slave 3

장점:
  - 쓰기 성능 N배 (Master N대)
  - 메모리 N배 (데이터 분산)
  - 자동 Failover (내장)
  - 수평 확장 가능

단점:
  - 복잡함
  - 설정 어려움
  - 일부 명령 제한 (MGET, 트랜잭션 등)
```

---

## 데이터 분산 방식: Hash Slot

**핵심: 데이터를 어떻게 나눠서 저장하나?**

### Hash Slot 개념

Redis Cluster는 **16,384개의 슬롯(Slot)** 으로 데이터를 나눈다.

```
전체 슬롯: 0 ~ 16383 (총 16,384개)

Master 3대면:
  Master 1: 슬롯 0 ~ 5460 (약 5461개)
  Master 2: 슬롯 5461 ~ 10922 (약 5461개)
  Master 3: 슬롯 10923 ~ 16383 (약 5462개)
```

**각 Master가 슬롯의 일부를 담당한다.**

---

### 키가 어느 슬롯에 저장되나?

**해시 함수로 계산:**

```
슬롯 번호 = CRC16(key) % 16384

CRC16는 문자열을 숫자로 변환하는 함수다

예:
  key = "user:1000"
  CRC16("user:1000") = 31,329
  31,329 % 16,384 = 14,945
  
  → 슬롯 14,945에 저장
  → Master 3이 담당 (10923~16383)
  → Master 3에 저장
```

**자동으로 어느 Master에 저장할지 결정된다.**

---

### Hash Tag로 같은 슬롯에 저장

**문제:**

```
user:1000과 user:1000:profile을 같은 Master에 저장하고 싶다.

user:1000 → 슬롯 14,945 (Master 3)
user:1000:profile → 슬롯 3,756 (Master 1)

→ 다른 Master에 저장됨
→ MGET 같은 명령 안 됨
```

**해결: Hash Tag 사용**

```
중괄호 {} 안의 내용만 해시 계산:

user:{1000} → {1000} 부분만 해시
user:{1000}:profile → {1000} 부분만 해시

→ 둘 다 같은 슬롯
→ 같은 Master에 저장
```

**예시:**

```bash
SET user:{1000} "철수"
SET user:{1000}:profile "{"age":30}"
SET user:{1000}:orders "[1,2,3]"

모두 {1000}으로 계산:
  CRC16("1000") = 20,021
  20,021 % 16,384 = 3,637
  → Master 1에 모두 저장

이제 가능:
  MGET user:{1000} user:{1000}:profile
```

### 2. 슬롯 계산

### 3. MOVED 응답

**Master 1 → 클라이언트:**

```
(error) MOVED 14945 192.168.1.103:6379

의미:
  "슬롯 14,945는 192.168.1.103:6379에 있어"
  "거기로 가"
```

MOVED는 슬롯이 영구적으로 이동했다는 걸 의미한다

14945 슬롯은 192.168.1.103:6379로 이동됐으니 이쪽으로 리다이렉트 하겠다는 의미

ASK라는 응답도 있는데 이건 슬롯이 마이그레이션 중이다라는 뜻으로 마이그레이션 중이니 임시로 다른 곳으로 리다이렉트하겠다는 의미다

이게 무슨 소리냐면

슬롯 4가 추가되어서 슬롯 3의 일부가 슬롯 4로 마이그레이션 중이라고 해보자

그럼 아직 완전히 이동된게 아니니 찾으려는 데이터가 슬롯 3에 있을 수도, 슬롯 4에 있을 수도 있다

요청이 들어왔을 때 ASK 응답은, 찾는 데이터가 슬롯 3에 없다면 슬롯 4에 있을테니 그쪽으로 리다이렉트 해주겠다는 의미다

만약 슬롯 3에 있다면 ASK 응답 없이 그냥 반환한다

MOVED는 이동이 완전히 끝난 상태

**그렇다면 뭐하러 굳이 MOVED를 쓸까?**

이동이 끝났으니 그냥 바로 옮긴 슬롯으로 연결하면 될텐데 굳이 MOVED라는 응답을 보내야할 필요가 있을까?

클라이언트가 모를 수도 있기 때문이다

마이그레이션을 완료했다 하더라도 각 클라이언트가 슬롯 맵을 언제 업데이트할지 모른다

슬롯 맵은 간단히 말하면 슬롯이 어느 Master에 있는지 기록한 캐시 테이블이다

클라이언트에서 관리하기 때문에 레디스에서 마이그레이션을 했다해도 바로 슬롯 맵에 업데이트가 안될 수도 있다

서버가 일일이 통보하지 못하기 때문

따라서 슬롯 맵이 완전히 업데이트 되기 전까지 이동했다라는걸 알려주기 위한 응답이다

## Resharding (리샤딩)

**Master를 추가하거나 제거할 때 슬롯을 재분배한다.**

### Master 추가 시나리오

**초기 상태: Master 3대**

```
Master 1: 슬롯 0~5460 (5,461개)
Master 2: 슬롯 5461~10922 (5,461개)
Master 3: 슬롯 10923~16383 (5,462개)
```

**Master 4를 추가:**

```
목표:
  Master 1: 슬롯 0~4095 (4,096개)
  Master 2: 슬롯 4096~8191 (4,096개)
  Master 3: 슬롯 8192~12287 (4,096개)
  Master 4: 슬롯 12288~16383 (4,096개)
```

---

### Resharding 과정

**1. Master 4를 Cluster에 추가**

```
redis-cli --cluster add-node 192.168.1.104:6379 192.168.1.101:6379

Master 4:
  Cluster 참여
  슬롯 0개 (아직 할당 안 됨)
```

---

**2. 슬롯 재분배**

```
redis-cli --cluster reshard 192.168.1.101:6379

질문:
  "몇 개 슬롯을 옮길까요?" → 4096
  "어디서 가져올까요?" → all (모든 Master에서)
  "어디로 옮길까요?" → Master 4 ID

시작:
  각 Master에서 약 1,365개씩 슬롯 이동
  Master 1 → Master 4: 1,365개
  Master 2 → Master 4: 1,365개
  Master 3 → Master 4: 1,366개
```

---

**3. 슬롯 이동 과정**

**슬롯 하나 이동:**

```
예: 슬롯 12,288을 Master 3 → Master 4로

1. Master 3을 "migrating" 상태로
   CLUSTER SETSLOT 12288 MIGRATING Master4_ID

2. Master 4를 "importing" 상태로
   CLUSTER SETSLOT 12288 IMPORTING Master3_ID

3. 슬롯 12,288의 모든 키를 Master 4로 복사
   CLUSTER GETKEYSINSLOT 12288 100
   → 키 목록 받음
   MIGRATE 192.168.1.104 6379 key1 0 5000
   MIGRATE 192.168.1.104 6379 key2 0 5000
   ...

4. 모든 키 이동 완료

5. 슬롯 소유권 변경
   Master 3: CLUSTER SETSLOT 12288 NODE Master4_ID
   Master 4: CLUSTER SETSLOT 12288 NODE Master4_ID

6. 다른 모든 노드에 전파
   CLUSTER SETSLOT 12288 NODE Master4_ID
```

**이 과정을 4,096개 슬롯에 대해 반복!**

---

**4. 완료**

```
Master 1: 슬롯 0~4095
Master 2: 슬롯 4096~8191
Master 3: 슬롯 8192~12287
Master 4: 슬롯 12288~16383

데이터가 4대로 분산됨
쓰기 성능 4배!
```

---

### Resharding 중 서비스는?

**계속 동작한다!**

```
슬롯 12,288 이동 중:

클라이언트 → Master 3: GET key_in_12288

Master 3:
  키가 아직 내게 있으면: 바로 반환
  키가 이미 옮겨졌으면: ASK 응답
  
ASK 192.168.1.104:6379

클라이언트 → Master 4:
  ASKING (일시적 요청임을 알림)
  GET key_in_12288
  
Master 4:
  데이터 반환

→ 서비스 중단 없음
```

마이그레이션을 진행하면서 ASK 응답으로 데이터가 옮긴 곳으로 계속 리다이렉트 해주기 때문에 서비스를 멈추지 않아도 된다

## Cluster Failover

**Master가 죽으면 Slave가 자동으로 Master가 된다.**

Sentinel 없이도 Cluster 자체에 Failover 기능이 내장되어 있다.

### Failover 과정

**초기 상태:**

```
Master 1 - Slave 1
Master 2 - Slave 2
Master 3 - Slave 3
```

**Master 2 다운:**

```
Master 2 죽음!
  ↓
다른 Master들이 감지
  ↓
Master 1: "Master 2 응답 없음"
Master 3: "Master 2 응답 없음"
  ↓
과반수(2/3) 동의
  ↓
Master 2의 Slave 2가 자동 승격
  ↓
Slave 2 → Master 2로 변경
  ↓
슬롯 5461~10922 계속 담당
  ↓
서비스 계속
```

**새 구조:**

```
Master 1 - Slave 1
Master 2 (Slave 2에서 승격) - Slave 없음
Master 3 - Slave 3
```

---

### 죽었던 Master 2가 다시 살아나면?

```
옛날 Master 2 재부팅
  ↓
Cluster 참여
  ↓
"내가 Master였는데..."
  ↓
다른 노드들:
"지금 Master는 새 Master 2야"
"너는 Slave가 돼"
  ↓
Slave로 전환
  ↓
새 Master 2의 Slave가 됨
```

**최종:**

```
Master 1 - Slave 1
Master 2 (새) - Slave (옛날 Master 2)
Master 3 - Slave 3
```

### Java 코드

```java
@Configuration
public class RedisConfig {
    
    @Bean
    public RedisConnectionFactory redisConnectionFactory() {
        RedisClusterConfiguration clusterConfig = 
            new RedisClusterConfiguration(Arrays.asList(
                "192.168.1.101:6379",
                "192.168.1.102:6379",
                "192.168.1.103:6379",
                "192.168.1.104:6379",
```

## Cluster 제약사항

### 1. 다중 키 명령 제한

**불가능:**

```bash
# 키들이 다른 슬롯에 있으면 에러
MGET user:1000 user:2000 order:5000

(error) CROSSSLOT Keys in request don't hash to the same slot
```

MGET은 여러 개의 키를 한 번에 가져오는 명령어다

그런데 

```
user:1000 → 슬롯 14945 → Master 3
user:2000 → 슬롯 8482 → Master 2
order:5000 → 슬롯 4314 → Master 1
```

서로 다른 슬롯에 있기 때문에 한 번에 가져올 수가 없다

**가능:**

```bash
# Hash Tag로 같은 슬롯에 저장
MGET user:{group1}:1000 user:{group1}:2000 order:{group1}:5000

→ 모두 같은 슬롯
→ 같은 Master
→ 성공
```

---

### 2. 트랜잭션 제한

**불가능:**

```bash
MULTI
SET user:1000 "철수"  # 슬롯 14,945
SET user:2000 "영희"  # 슬롯 8,482
EXEC

(error) CROSSSLOT
```

이것 역시 다른 슬롯에 있기 때문에 발생하는 문제다

MULTI 명령어는 하나의 트랜잭션으로 묶어서 시작한다는 명령어고

EXEC은 트랜잭션을 실행한다는 명령어다

그런데 명령어들이 다 다른 슬롯에 있기 때문에 하나의 트랜잭션으로 묶을 수 없어서 에러가 발생한다

**가능:**

```bash
MULTI
SET user:{1000} "철수"
SET profile:{1000} "..."
EXEC

→ 같은 슬롯
→ 성공
```

---

### 3. SELECT 명령 불가

```bash
SELECT 1

(error) ERR SELECT is not allowed in cluster mode
```

Cluster는 DB 번호 개념이 없다. 모두 DB 0만 사용.

이게 무슨 소리냐면

레디스에서는 테이블은 없지만 DB는 있다

그런데 이 DB라는건 그냥 Key의 목적에 따라 분류하는 네임스페이스다

```
DB = 키를 목적별로 분류하는 네임스페이스

DB 0: 캐시
DB 1: 세션
DB 2: 임시 데이터

→ 같은 키 이름이어도 DB 다르면 별개
```

그래서 Redis에서 SELECT 명령어로 해당 네임스페이스로 전환해서 사용이 가능했는데

레디스 클러스터에는 이 DB 네임스페이스 기능이 없다

그리고 레디스의 DB 기능은 잘 안쓰는 기능이니 참고

---

### 4. Pub/Sub 제한

**전역 Pub/Sub 불가:**

```
일반 Redis:
  PUBLISH channel message
  → 모든 구독자에게 전달

Cluster:
  PUBLISH channel message
  → 같은 노드의 구독자에게만 전달
```

레디스에도 카프카 처럼 pub/sub 기능이 있다

하지만 카프카처럼 복잡한 기능을 제공하지는 않는다

휘발성이라는 특성 때문에 메시지를 따로 저장하지는 않고 들어온 메시지를 구독자에게 전달하기만 한다

**해결:**

```bash
# Cluster 전체로 전파
PUBLISH {channel} message

Hash Tag 사용하면 같은 슬롯
→ 같은 Master
→ 해당 Master 구독자들에게 전달
```

**좋은 예:**

```
Master 2: 슬롯 5,461개
Master 3: 슬롯 5,462개

→ 균등 분산
```

---

### 2. Hash Tag 남용 금지

**나쁜 예:**

```
모든 키에 같은 Hash Tag:

user:{app}:1000
user:{app}:2000
order:{app}:5000
...

→ 모두 같은 슬롯
→ 한 Master에 몰림
→ Cluster 의미 없음
```

**좋은 예:**

```
그룹별로 다른 Hash Tag:

user:{tenant1}:1000
user:{tenant2}:2000
order:{tenant3}:5000

→ 여러 슬롯에 분산
```

---

### 3. 노드 수는 홀수로

**이유:**

```
Master 2대:
  1대 죽으면 과반수 못 채움
  → Failover 불가

Master 3대:
  1대 죽어도 2대 살아있음
  → 과반수 (2/3)
  → Failover 가능

Master 4대:
  2대 죽으면 과반수 못 채움
  → 3대와 동일한 내결함성
  → 비용만 증가

Master 5대:
  2대 죽어도 3대 살아있음
  → 과반수 (3/5)
  → Failover 가능
```

**권장: 3, 5, 7대**

## Cluster vs Sentinel 비교

| 항목 | Master-Slave + Sentinel | Cluster |
|------|-------------------------|---------|
| 쓰기 성능 | 1대 한계 | N배 확장 |
| 메모리 | 1대 한계 | N배 확장 |
| 데이터 분산 | 불가 | 가능 |
| Failover | 자동 (Sentinel) | 자동 (내장) |
| 구성 복잡도 | 중간 | 높음 |
| 다중 키 명령 | 제한 없음 | 제한 있음 |
| 최소 서버 수 | 3대 (M+S+S) | 6대 (M3+S3) |
| 사용 사례 | 중소규모 | 대규모 |

---

## 언제 Cluster를 쓸까?

### Cluster가 필요한 경우

```
1. 쓰기 성능이 부족할 때
   Master 1대로 감당 안 될 때

2. 메모리가 부족할 때
   데이터가 1대에 안 들어갈 때

3. 수평 확장이 필요할 때
   트래픽 증가에 대비

4. 대규모 서비스
   MAU 100만 이상
```

---

### Sentinel이면 충분한 경우

```
1. 쓰기 성능 충분
   Master 1대로 처리 가능

2. 메모리 충분
   데이터가 1대에 들어감

3. 중소규모 서비스
   MAU 100만 미만

4. 다중 키 명령 많이 씀
   MGET, MSET, 트랜잭션 등
```

## 실전 예시

### 사용자 세션 저장

**Sentinel로 충분:**

```java
// 세션 저장 (쓰기 적음)
session.setAttribute("user", user);

// 세션 조회 (읽기 많음)
User user = session.getAttribute("user");

→ 읽기 >> 쓰기
→ Slave로 읽기 분산
→ Sentinel 구성 적합
```

---

### 실시간 랭킹

**Cluster 필요:**

```java
// 점수 업데이트 (쓰기 많음)
redis.zadd("ranking", score, userId);

// 상위 10명 조회
redis.zrevrange("ranking", 0, 9);

사용자 1000만 명:
  → 초당 점수 업데이트 10만 건
  → Master 1대로 부족
  → Cluster로 분산 필요
```

---

### 캐시

**Sentinel로 충분:**

```java
// 캐시 저장
cache.put("product:1000", product);

// 캐시 조회
Product p = cache.get("product:1000");

→ 읽기 >> 쓰기
→ 캐시 미스만 쓰기
→ Sentinel 구성 적합
```

## 정리

### Cluster란?

```
여러 대의 Master로 데이터를 분산 저장
쓰기 성능과 메모리를 수평 확장
```

### 핵심 개념

```
Hash Slot:
  16,384개 슬롯
  키마다 슬롯 계산
  각 Master가 슬롯 일부 담당

데이터 분산:
  Master 3대면 각각 1/3씩
  쓰기 성능 3배
  메모리 3배
```

### 장점

```
- 쓰기 성능 N배
- 메모리 N배
- 자동 Failover
- 수평 확장 가능
```

### 단점

```
- 설정 복잡
- 다중 키 명령 제한
- 트랜잭션 제한
- 최소 6대 필요
```

### 선택 기준

```
Sentinel:
  - 중소규모
  - Master 1대로 충분
  - 다중 키 명령 많음

Cluster:
  - 대규모
  - Master 1대로 부족
  - 수평 확장 필요
```
