# 들어가며

카프카를 여러 프로젝트에서 써왔지만, 정작 정확한 이론에 대해서는 제대로 파악하지 못하고 쓰는 것 같다는 느낌을 받았다.

비동기 방식으로, 브로커를 통해 메시지를 주고받는 기능이라는 건 알지만 어째서? 왜? 이런 기능이 필요한건지에 대한 이유를 알지 못한채 사용하고 있었다.

이러한 의문을 해소하기 위해 파고들며 공부한 내용을 정리하려고 한다.

# 카프카란?

아파치 카프카는 실시간으로 레코드의 스트림을 게시, 구독, 저장, 처리할 수 있는 분산 데이터 스트리밍 플랫폼이다

어려운 단어들이 난무하는데 쉽게 풀어서 설명하자면..

## 프로듀서와 컨슈머

카프카에서는 프로듀서와 컨슈머라는 두 가지 주요 역할이 있다

카프카는 기본적으로 메시지를 주고받는 시스템이다

이때 메시지를 보내는 쪽을 프로듀서
메시지를 받는 쪽을 컨슈머라고 한다

쉬운 예시를 들자면

```
프로듀서 = 영상 업로드하는 유튜버
컨슈머 = 구독자들
```

영상이 올라가면 여러 구독자가 각자 시간에 맞춰 볼 수 있는 것처럼
카프카도 메시지를 저장해두고 여러 컨슈머가 각자 필요할 때 가져가서 처리한다

## 레코드

먼저 레코드란 카프카에서 가장 작은 데이터 단위를 말하는데, 쉽게 얘기하면 "한건의 메시지"다

구조는 보통
- key -> byte[]
- value -> byte[]
- timestamp
- 기타 메타데이터

로 구성되어 있는데 JSON을 떠올리면 쉽다 즉, JSON 하나가 레코드 1개라고 이해하면 쉽다

레코드는 key-value 형태의 한 건의 메시지다. value에는 JSON이 들어갈 수도 있고, 문자열, 숫자, 바이너리 등 뭐든 올 수 있다. 대부분의 실무에서는 JSON을 많이 쓰기 때문에 '레코드 = JSON 하나'처럼 이해해도 크게 문제없다

```
key: 123
value: {
  "userId": 123,
  "action": "order_created",
  "amount": 15000
}
timestamp: 1710000000000

-> 레코드 1개
```

## 스트림

그리고 스트림은 이런 레코드들의 연속 흐름이다

더 정확히 말하자면, "시간 순서대로 이어지는 데이터의 흐름"이다

서버 로그로 예시를 들어보자

```
[12:00:01] GET /users
[12:00:02] POST /login
[12:00:03] GET /products
[12:00:04] POST /orders
```

각각 로그로 기록된 이벤트를 하나의 레코드로 본다면, 이렇게 시간순으로 이어진 레코드들을 스트림이라고 부른다

스트림에서 가장 중요한 점은 "시간 순서"와 "그 순서가 의미를 가지는 흐름"이다

카프카는 개별 레코드뿐만 아니라, 시간 순서대로 이어지는 데이터의 흐름을 중요하게 다루는 시스템이다

또한 순서는 전체가 아니라 같은 흐름(파티션) 내에서만 보장된다

## 파티션은 또 뭐고, 왜 같은 파티션내에서만 순서를 보장할까?

우선 파티션을 이해하기 전에 토픽에 대한 개념부터 이해하고 가야 한다

토픽은 카프카에서 메시지를 분류하는 논리적 단위다

쉽게 말하면 "메시지 카테고리" 또는 "메시지 보관함"이라고 생각하면 된다

데이터베이스의 테이블처럼, 카프카에서 메시지를 저장할 때 "어디에 저장할지"를 정하는 이름이 바로 토픽이다

```
예시)

order-events  -> 주문 관련 이벤트만 저장
payment-events  -> 결제 관련 이벤트만 저장
```

만약 토픽이 없다면 모든 메시지가 하나의 거대한 큐에 뒤섞여 들어간다.

주문, 결제, 알림, 로그가 전부 한 곳에 쌓이면 각 서비스가 자기에게 필요한 메시지를 찾기 위해 전체를 일일이 확인해야 하는 문제가 생긴다

<br>

**그렇다면 파티션은 뭘까?**

파티션은 토픽을 물리적으로 나눈 저장 단위다.

하나의 토픽은 1개 이상의 파티션으로 구성되고, 각 파티션은 독립적인 큐처럼 동작한다

쉽게 이해할만한 예시로는

```
토픽 = 은행 창구
파티션 = 1번 창구, 2번 창구, 3번 창구
```

고객(메시지)이 오면 특정 창구(파티션)로 배정되어 줄을 선다
각 창구는 독립적으로 처리되므로 병렬로 빠르게 처리할 수 있다

실제 구조는 다음과 같다

```
order-events 토픽
|_ 파티션 0 : [메시지 A, 메시지 B, 메시지 C]
|_ 파티션 1 : [메시지 D, 메시지 E, 메시지 F]
```

파티션이 1개라면 모든 메시지가 하나의 큐에 순차적으로 쌓이고, 컨슈머도 하나씩 처리해야 하지만
만약 1만 건의 메시지가 들어오거나 더 많은 메시지가 들어오게 되면? 병목이 생긴다

파티션이 여러개 있다면 병렬 처리가 가능해져 이런 병목을 해소할 수 있다

<br>

여기선 내가 오해했던 부분인데..

"파티션을 여러 개로 나누면 순서가 깨지는 거 아냐?"

예를 들어, 주문 → 결제 프로세스에서 2개의 메시지가 발생한다고 생각해보자:
```
- 메시지 1: `status를 Ready로 바꿔!`
- 메시지 2: `status를 Paid로 바꿔!`
```

나는 이렇게 될까봐 걱정했다:

```
파티션 0: [Ready 메시지]
파티션 1: [Paid 메시지]
```

만약 컨슈머가 파티션 1부터 읽으면 Paid가 먼저 처리되고, Ready가 나중에 처리되는 거 아냐? 
순서가 완전히 망가지는거 아닌가?

그런데 카프카는 key의 해시값으로 파티션을 결정한다는 점을 간과하고 있었다..

프로듀서가 메시지를 보낼 때 key를 지정하면, **같은 key를 가진 메시지는 항상 같은 파티션**으로 들어간다

<br>

**그렇다면 왜 같은 파티션 내에서만 순서가 보장될까?**

카프카는 파티션 단위로 메시지를 저장하고, 각 파티션은 독립적으로 병렬처리된다

이 때문에 전체 순서는 보장되지 않고, 같은 파티션 내에서만 순서가 보장된다

예시를 보자

주문 이벤트가 3개 들어왔다고 가정하다

```
1. userId = 123, OrderCreated
2. userId = 456, OrderCreated
3. userId = 123, PaymentCompleted
```

카프카는 메시지를 파티션에 분배할때 key 해시값을 기준으로 나누는데 key를 userId로 설정했다고 했을때

```
key = 123 -> 파티션 0
key = 456 -> 파티션 1
```

파티션 0에는

```
[1. OrderCreated (userId=123)]
[3. PaymentCompleted (userId=123)]
```

파티션 1에는

```
[2. OrderCreated (userId=456)]
```

이 저장되어 있을 것이다

이때, 전체 순서는 컨슈머가 언제 읽느냐에 따라 달라진다

컨슈머 A가 먼저 읽는다면 1 -> 3 - > 2
컨슈머 B가 먼저 읽는다면 2 -> 1 -> 3

이렇게 전체 순서는 보장이 되지 않는다

하지만, 같은 userId의 이벤트는 항상 같은 파티션에 들어가기 때문에
userId = 123의 이벤트는 항상 OrderCreated -> PaymentCompleted 순서가 보장된다

## 시간 순서와 데이터의 흐름을 왜 중요시할까?

카프카는 "최종 상태"가 아니라 "상태를 만든 이벤트들의 순서"를 저장한다

코드 느낌으로 보자

```
order.setStatus(PAID);
```

이런식으로 되어 있을 때, 이전에 어떤 이벤트가 있었는지 알 수 없기 때문에 버그가 나면 원인을 추적하기 어렵다

그런데 이벤트를 순서대로 저장했다면?

```
OrderCreated
PaymentRequested
PaymentCompleted
```

주문 생성 -> 결제 요청 -> 결제 완료라는 흐름을 파악할 수 있어서 디버깅이 쉽고, 언제든 다시 이 순서대로
데이터를 읽을 수 있다

또 여기서 왜 "순서"가 필수인지 알 수 있다

```
PaymentCompleted
OrderCreated
```

이건 말이 안되는 상태다. 주문을 생성하지도 않았는데 결제 완료가 되었기 때문

따라서 이 "흐름"이 카프카에서는 굉장히 중요한 개념이다

```
주문 서비스 → 이벤트 발생
             ↓
알림 서비스 → 메시지 전송
             ↓
통계 서비스 → 집계
             ↓
추천 시스템 → 사용자 분석
```

그리고 이런 흐름을 "같은 이벤트 스트림"을 구독해서 각자 처리할 수가 있게된다

## 컨슈머 그룹

파티션 3개짜리 토픽이 있고, 처리 속도를 높이려고 컨슈머를 5개 띄웠다고 가정해보자

**컨슈머는 기본적으로 구독한 토픽의 모든 파티션을 읽으려고 시도한다**

```
order-events 토픽
|_ 파티션 0: [A, B, C]
|_ 파티션 1: [D, E, F]  
|_ 파티션 2: [G, H, I]

컨슈머 5개가 각자 읽으면?
├─ 컨슈머 1 → 파티션 0, 1, 2 전부 읽음
├─ 컨슈머 2 → 파티션 0, 1, 2 전부 읽음
├─ 컨슈머 3 → 파티션 0, 1, 2 전부 읽음
├─ 컨슈머 4 → 파티션 0, 1, 2 전부 읽음
└─ 컨슈머 5 → 파티션 0, 1, 2 전부 읽음
```

이렇게 되면 문제가, 같은 메시지를 5개의 컨슈머가 중복 처리하게 된다
주문은 1건인데 결제를 5번 처리하는 참사가 발생할 수 있다

이런 문제를 해결하기 위해 같은 그룹의 컨슈머들은 파티션을 나눠서 처리하자는 개념이 등장했다

```
컨슈머 그룹 "order-processors"
├─ 컨슈머 1 → 파티션 0만 담당
├─ 컨슈머 2 → 파티션 1만 담당
└─ 컨슈머 3 → 파티션 2만 담당

각 메시지는 그룹 내에서 딱 1번만 처리됨
3개 파티션을 3개 컨슈머가 병렬 처리
```

즉, 컨슈머 그룹은 같은 로직을 가진 컨슈머들을 묶는 이름이다

컨슈머 그룹의 핵심 규칙은 "같은 그룹에 속한 컨슈머들은 파티션을 나눠 가진다"

1개 파티션 = 같은 그룹 내 1개 컨슈머만 처리 가능하다
1개 컨슈머 = 여러 파티션이 담당 가능하다

즉, 파티션 하나는 딱 하나의 컨슈머만 처리할 수 있지만
컨슈머는 여러 파티션을 동시에 담당할 수 있다

실제 코드를 보며 알아보자

@KafkaListener(topics = "pay-events.public.pay_outbox", groupId = "order-service-pay-group")
public void onPayEvent(String message) {
  ...
}

위 코드는 결제 완료 또는 취소 메시지를 전달받아서 주문 상태를 업데이트하는 컨슈머다.

여기서 groupId = "order-service-pay-group" 이 부분이 바로 컨슈머 그룹이다.

만약 order-service를 서버 3대로 띄웠는데 컨슈머 그룹을 설정하지 않거나 각자 다른 그룹으로 설정하면,
3대 서버가 같은 결제 메시지를 각각 처리하게 된다.

예를 들어 "주문#123 결제 완료" 메시지 1개가 들어오면:

```
- 서버 A가 처리 → 포인트 1000원 적립
- 서버 B가 처리 → 포인트 1000원 적립
- 서버 C가 처리 → 포인트 1000원 적립
```

→ 같은 주문에 포인트가 3000원 적립되는 참사 발생

그런데 같은 컨슈머 그룹(order-service-pay-group)으로 묶으면,
카프카가 자동으로 파티션을 분배해서 같은 메시지를 딱 1번만 처리하게 보장한다.

```
결제 완료 메시지 1개 들어옴

서버 A만 → 포인트 1000원 적립 (1번만)
서버 B → 안 함
서버 C → 안 함

→ 같은 메시지를 1번만 처리
```

<br>

**또 내가 이해못했던점 정리...**

컨슈머 A가 파티션 0과 4를 담당하고 있다고 해보자

메시지는 주문#123 형태로 들어왔다고 해보자

이랬을 때 나는 그럼 컨슈머는 자기가 담당하는 모든 메시지를 읽기 때문에 파티션 0과 4를 읽어서

결제는 userId=123에서만 했는데 userId=456도 주문 상태가 업데이트 되는거 아닌가?같은 생각을 했다..

정말 바보같은 생각인게 애초에 userId=456에서 주문을 하지 않았다면 파티션 4는 비어있어서 주문 상태를 업데이트 하지 못한다

## 오프셋

오프셋이란 파티션 내 각 메시지의 위치를 나타내는 번호다

파티션에 저장된 메시지마다 0부터 시작하는 고유한 번호가 붙어있다

```
파티션 0:
  오프셋 0: [메시지A]
  오프셋 1: [메시지B]
  오프셋 2: [메시지C]
  오프셋 3: [메시지D]
  오프셋 4: [메시지E]
```

오프셋은 책갈피라고 생각하면 쉽다
책갈피의 용도는 내가 어디까지 읽었는지를 표시하기 위함인데, 오프셋도 마찬가지로 컨슈머가 메시지를 어디까지 읽었는지 표시하기 위함이다

**만약 오프셋이 없다면?**

```
파티션 0: [메시지1, 메시지2, 메시지3, 메시지4, 메시지5]

컨슈머 A가 읽는 중:
메시지1 읽음
메시지2 읽음
메시지3 읽음

→ 서버 재시작 💥

다시 시작했을 때:
"어디까지 읽었더라...?"
```

이렇게 됐을 때 카프카는 2가지 선택지가 있는데

1. 처음부터 다시 읽기 -> 중복 처리됨
2. 대충 건너뛰기 -> 메시지 유실됨

둘다 성능면에서나 데이터 정합성면에서나 문제가 발생한다

이러한 문제를 오프셋으로 해결할 수 있다

컨슈머가 메시지를 처리하고 나서 "오프셋 N까지 읽었어"라고 카프카에 저장한다

```
파티션 0:
오프셋 0: [메시지1]
오프셋 1: [메시지2]
오프셋 2: [메시지3]
오프셋 3: [메시지4]
오프셋 4: [메시지5]

컨슈머 A:
메시지1 처리 → 오프셋 0 커밋
메시지2 처리 → 오프셋 1 커밋
메시지3 처리 → 오프셋 2 커밋

→ 서버 재시작 💥

다시 시작:
카프카: "너 마지막으로 오프셋 2까지 커밋했네?"
컨슈머: "그럼 오프셋 3부터 읽을게"

→ 메시지4부터 이어서 처리
```

**그럼 이 오프셋은 어디에 저장될까?**

카프카의 내부 토픽인 **__consumer_offsets**에 저장된다

## 브로커

브로커란 카프카 메시지를 실제로 저장하고 관리하는 서버다

지금까지 우리가 "카프카"라고 추상적으로 얘기했던게 바로 브로커 서버들이다

**만약 브로커가 없다면 어떻게 될까?**

프로듀서와 컨슈머가 직접 연결되어 있다고 생각해보자

```
프로듀서 → 컨슈머

문제점:
- 컨슈머가 꺼져있으면? 메시지 유실
- 컨슈머가 느리면? 프로듀서가 기다려야 함
- 컨슈머가 여러 개면? 프로듀서가 일일이 전송해야 함
```

비효율적이고 메시지 유실 위험이 있다

**브로커가 있다면?**

브로커가 중간에서 메시지를 저장하고 중개해준다

```
프로듀서 → [브로커] → 컨슈머

브로커의 역할:
- 프로듀서가 보낸 메시지 저장
- 컨슈머가 요청하면 메시지 전달
- 컨슈머가 꺼져있어도 메시지 보관
```

프로듀서는 일단 브로커에 던지고 끝이다
컨슈머는 준비되면 브로커에서 가져오기만 하면 된다

이런식으로 비동기 처리가 가능해진다

브로커는 물리적인 서버로 여러 대의 브로커를 카프카 클러스터라고 부른다

브로커는 보통 여러 대를 쓰는데 그 이유는

1. 장애 대응에 용이하다
```
브로커 1대만 있을 때:
브로커 죽음 → 전체 시스템 마비

브로커 3대 있을 때:
브로커 1 죽음 → 브로커 2, 3이 대신 처리
```

2. 성능 향상을 기대할 수 있다

```
브로커 1대: 초당 1000개 메시지 처리
브로커 3대: 초당 3000개 메시지 처리
```

파티션을 여러 브로커에 분산시켜서 병렬 처리가 가능해져 성능이 향상될 수 있다

각 파티션은 특정 브로커에 저장되므로 파티션을 여러 브로커에 분산시켜서 부하를 분산할 수 있다

```
토픽: order-events (파티션 3개)

파티션 0 → 브로커 1에 저장
파티션 1 → 브로커 2에 저장
파티션 2 → 브로커 3에 저장
```

**그런데 다른 토픽들도 파티션을 가지고 있을텐데 브로커는 어떻게 구분해서 저장할까?**

브로커가 파티션을 저장할 때 토픽명 + 파티션 번호로 구분해서 저장한다

```
브로커 1에 저장된 파티션들:

order-events-0       (order-events 토픽의 파티션 0)
order-events-1       (order-events 토픽의 파티션 1)
pay-events-0         (pay-events 토픽의 파티션 0)
notification-events-2 (notification-events 토픽의 파티션 2)
```

실제 디스크 저장 구조를 보면 이런식이다

```
/var/lib/kafka/data/
├─ order-events-0/
│  ├─ 00000000000000000000.log (메시지 파일)
│  └─ 00000000000000000000.index (인덱스 파일)
│
├─ order-events-1/
│  ├─ 00000000000000000000.log
│  └─ 00000000000000000000.index
│
├─ pay-events-0/
│  ├─ 00000000000000000000.log
│  └─ 00000000000000000000.index
│
└─ notification-events-2/
   ├─ 00000000000000000000.log
   └─ 00000000000000000000.index
```

각 파티션마다 별도의 폴더가 생기고, 그 안에 메시지가 저장되는 방식이다

```
브로커 1:
- order-events-0
- pay-events-1
- notification-events-0

브로커 2:
- order-events-1
- pay-events-2
- notification-events-1

브로커 3:
- order-events-2
- pay-events-0
- notification-events-2
```

이렇게 서로 다른 토픽의 파티션들이 여러 브로커에 섞여서 저장된다

## Zookeeper

카프카를 띄우기 위해서는 주키퍼가 필수로 설치되어 있어야 한다

주키퍼란 카프카 클러스터의 메타데이터를 관리하는 코디네이터다

설명이 좀 난해하고 어려운데, 쉽게 설명하자면

카프카의 정보를 저장하고 관리하는 별도의 저장소다
카프카 브로커들이 돌아가려면 필요한 정보들이 있는데, 그걸 주키퍼가 저장하고 있다

주키퍼가 저장하는 정보는 다음과 같다

1. 브로커 목록

```
현재 살아있는 브로커:
- 브로커 1 (IP: 192.168.1.10)
- 브로커 3 (IP: 192.168.1.12)

죽은 브로커:
- 브로커 2 (IP: 192.168.1.11)
```

어떤 브로커가 살아있고 죽었는지를 기록한다

2. 토픽과 파티션 정보

```
토픽 목록:
- order-events: 파티션 3개
- pay-events: 파티션 5개

각 파티션 위치:
- order-events-0 → 브로커 1에 저장
- order-events-1 → 브로커 2에 저장
- order-events-2 → 브로커 3에 저장
```

어떤 토픽이 있고, 각 파티션이 어디 저장되어 있는지 기록한다

3. 리더 정보

```
파티션별 리더:
- order-events-0의 리더: 브로커 1
- order-events-1의 리더: 브로커 2
- order-events-2의 리더: 브로커 3
```

각 파티션의 리더가 누군지 기록한다

브로커 리더가 죽으면 주키퍼가 감지해서 새로운 리더를 자동으로 선출한다

**이 리더가 뭘 말하는걸까?**

리더라는건 특정 파티션에 대해 읽기/쓰기를 담당하는 브로커를 말한다

하나의 파티션은 여러 브로커에 복제될 수 있는데, 그중에서 실제로 일하는 브로커 1개를 리더라고 한다

왜 리더가 필요할까?

파티션이 여러 브로커에 복제되어 있는 상황을 보자

```
order-events-0 파티션이 3개 브로커에 복제됨:
- 브로커 1에 복사본
- 브로커 2에 복사본
- 브로커 3에 복사본
```

여기서 문제는 프로듀서가 메시지를 보낼 때 어느 브로커에 보내야 할까?

```
프로듀서:
"order-events-0에 메시지 쓸건데...
브로커 1? 브로커 2? 브로커 3? 어디에 써?"

만약 3개 브로커에 동시에 쓰면?
→ 순서 꼬임, 중복, 동기화 문제 발생
```

이럴때 리더를 지정해서 해당 브로커에게 메시지를 보내면 된다

복제를 해두는 이유는 보통 백업용으로 복제해둔다

4. 설정 정보

```
토픽 설정:
- order-events: 복제본 2개, 보관 기간 7일
- pay-events: 복제본 3개, 보관 기간 30일
```

<br>

**주키퍼가 필수인 이유를 정리해보면**

```
브로커 1 시작:
1. 주키퍼에 접속
2. "내가 담당할 파티션이 뭐지?" 확인
3. "다른 브로커들은 누가 살아있지?" 확인
4. 정보 받아서 작업 시작
```

따라서 주키퍼가 없다면 브로커가 필요한 정보를 가져오지 못해서 실행할 수가 없다

## 복제 (Replication)

복제란 같은 파티션의 데이터를 여러 브로커에 중복 저장하는 것을 말한다

즉, 하나의 파티션을 여러 브로커에 복사해서 저장한다

```
order-events-0 파티션 (복제본 3개):
- 브로커 1: 원본
- 브로커 2: 복사본
- 브로커 3: 복사본
```

**왜 복제가 필요할까?**

복제 없이 브로커 1대에만 저장했다고 해보자

```
order-events-0 파티션:
- 브로커 1에만 저장

브로커 1 죽음

→ order-events-0 데이터 전부 유실
→ 서비스 중단
```

데이터가 유실되거나, 서비스 중단 상황이 발생한다

만약 데이터를 복제해서 여러 브로커에 저장해둔다면 리더 브로커가 죽더라도 새로운 리더를 바로 선출하기 때문에 서비스가 중단되거나 데이터가 유실될 상황을 예방할 수 있다

복제 동작 방식은 다음과 같다

프로듀서가 메시지를 보낼 때

```
1. 프로듀서 → 리더(브로커 1)에 메시지 씀
2. 리더 → 팔로워들(브로커 2, 3)에 복제
3. 팔로워들이 복제 완료 응답
4. 프로듀서에게 "저장 완료" 응답
```

리더가 먼저 받은 후에 팔로워들이 복사한다

복사본에는 개수도 설정할 수 있다

복사본 개수는 토픽을 생성할 때 지정할 수 있다

```
kafka-topics.sh --create \
  --topic order-events \ # 토픽명
  --partitions 3 \ # 파티션 개수
  --replication-factor 3 \ # 복제본 개수
  --bootstrap-server localhost:9092
```

--replication-factor 다음에 원하는 복제본의 개수를 지정한다

복제본이 많을수록 안정성이 증가하지만

그만큼 저장공간을 사용하고 네트워크 부하가 증가하는 단점도 있기 때문에 상황에 따라 적절히 조절해서 써야한다

보통은 복제본 2~3개를 사용한다

## ISR (In-Sync Replicas)

ISR이란 리더와 데이터가 동기화된 팔로워 목록을 말한다

즉, 리더 브로커와 실시간으로 데이터를 복제하고 있는 팔로워들의 리스트다

```
order-events-0 파티션 (복제본 3개):
- 브로커 1: 리더
- 브로커 2: 팔로워 (ISR)
- 브로커 3: 팔로워 (ISR)

ISR = [브로커 1, 브로커 2, 브로커 3]
```

**ISR이 왜 필요할까?**

팔로워가 느려진 상황이 있을 것이다

```
리더 브로커: [메시지1, 메시지2, 메시지3, 메시지4, 메시지5]

팔로워 A: [메시지1, 메시지2, 메시지3, 메시지4, 메시지5]
팔로워 B: [메시지1, 메시지2] (네트워크 느림, 서버 과부하)
```

만약 이 상황에서 리더가 죽으면 누구를 새 리더로 선출해야 할까?

당연히 리더와 최신 동기화된 팔로워 A를 리더로 선출해야 한다

카프카는 리더와 동기화된 팔로워만 ISR에 포함 시킨다

즉, 리더와 제대로 동기화되지 않은 팔로워 B는 ISR에 포함되지 않고 팔로워 A만 ISR에 포함되고
ISR에 포함된 유일한 팔로워 A가 새 리더로 선출된다

즉, ISR은 리더가 될 자격이 있는 브로커들의 목록이라고 보면 된다

물론 ISR에는 리더도 포함되어 있다는 점은 유의해야 한다

<br>

**그럼 ISR 리스트 중에 누구를 리더로 선출할까?**

ISR 리스트 중 첫 번째 브로커를 선택한다

정확히는, 가장 먼저 ISR에 추가된 브로커 또는 Replica 목록에서 가장 앞에 있는 살아있는 브로커를 선출한다

예시를 보면

```
order-events-0 파티션:
Replicas: [1, 2, 3]  ← 브로커 우선순위
ISR: [1, 2, 3]

브로커 1 (리더) 죽음

ISR에서 브로커 1 제외:
ISR: [2, 3]

Replicas 순서 확인:
[1(죽음), 2(살아있음), 3(살아있음)]
→ 2가 먼저니까 브로커 2를 새 리더로 선출
```

<br>

**그럼 Replicas 순서는 어떻게 정해질까?**

토픽을 생성할때 카프카가 자동으로 브로커를 배정한다

**Replicas랑 ISR 목록에서 가장 앞에 있는 브로커를 선택한다고 했는데 우선순위는?**

Replicas가 우선순위가 더 높다

Replicas는 기준 순서가 변하지 않지만, ISR은 제대로 동기화되지 않는 브로커가 있다면 제외시키므로 동적으로 변한다

따라서 Replicas 순서대로 확인하면서 ISR에 포함된 첫 번째 브로커를 새 리더로 선출한다

```
Replicas: [1, 2, 3]
ISR: [1, 3]  (브로커 2는 느려서 제외됨)

브로커 1 죽음:
1. Replicas 순서 확인: [1, 2, 3]
2. 1번? 죽음 X
3. 2번? ISR에 없음 X
4. 3번? ISR에 있음 O → 3번이 새 리더
```

## ACK (Acknowlegement)

ACK란 프로듀서가 메시지를 보낸 후 "저장 완료" 응답을 받는 방식이다

프로듀서가 카프카에 메시지를 보냈을 때
- 바로 다음 메시지를 보낼까?
- 저장 확인 후 보낼까?
- 복제까지 완료된 후 보낼까?

이걸 acks 설정으로 조정한다

<br>

**ACK의 설정 3가지**

1. acks = 0 (응답 안받음)

```
프로듀서 → 메시지 전송 → 끝
           ↓
        브로커 (응답 안 함)
```

특징
- 응답을 안기다림
- 가장 빠름
- 그러나 데이터 유실 가능성이 높음
- 로그나 모니터링 용으로 쓸때 유용하다(유실돼도 큰 문제 없는 데이터)

<br>

2. acks = 1 (리더만 확인)

```
프로듀서 → 메시지 전송 
           ↓
        리더 브로커 저장 완료 → "OK" 응답
           ↓
        팔로워들에 복제 (비동기)
```

특징
- 리더에 저장되면 응답한다
- 적당한 속도
- 그러나 리더가 죽으면 유실될 가능성이 높다(팔로워 복제 전에)
- 일반적인 메시지를 보낼때 쓰는 설정이다

<br>

3. acks = all (또는 -1) (모두 확인)

```
프로듀서 → 메시지 전송
           ↓
        리더 브로커 저장
           ↓
        ISR의 모든 팔로워 복제 완료
           ↓
        "OK" 응답 → 프로듀서
```

특징
- ISR 전부 복제 완료 후 응답한다
- 3가지 방식 중 가장 느리다
- 데이터 유실 가능성이 거의 없다
- 금융 데이터, 주문 정보 등 절대 유실되면 안되는 데이터를 다룰 때 쓴다

## 메시지 보관 기간

Retention이라고 하는데 카프카에 저장된 메시지를 얼마나 오래 보관할지를 설정한다

기본 보관 기간은 7일이고, 7일이 지난 메시지는 자동으로 삭제된다

```
# 토픽 생성 시
kafka-topics.sh --create \
  --topic order-events \
  --config retention.ms=604800000  # 7일 (밀리초)
```

기간뿐만 아니라 용량을 기준으로도 설정할 수 있다

```
--config retention.bytes=1073741824  # 1GB
```

이렇게 기간이나 용량을 설정하는 이유는 당연히 디스크 공간을 절약하기 위함이다

따라서 보통 실무에서는

```
로그 데이터: 1-3일 (빨리 삭제)
주문 데이터: 30일 (오래 보관)
감사 데이터: 1년 또는 영구 (법적 요구사항)
```

이런식으로 데이터 유형에 따라 기간을 설정한다
